# FX-RV32 确定性验证测试

> 用于论文中证明 CPU 指令执行的确定性——相同程序、相同输入、相同周期数、相同结果。每个测试重复执行 40 轮，每轮用 `csrr 0xB00`（mcycle CSR）测量周期数，存至 `[0x140]` 起始的结果数组。PASS 标志为 `[0xFC] = 0x00000000`。

## 一、中文版：测试介绍与结果

### 测试 6：前递确定性（test6_forwarding.S）

本测试通过构造一条由 10 条连续 addi 指令组成的深度数据依赖链来验证流水线前递通路的确定性。链中每条指令的源操作数 rs1 均为上一条指令的目标寄存器 rd，相邻指令之间不存在任何气泡，迫使 EX 阶段在每个周期都必须从 EX/MEM 流水线寄存器中获取前递数据。其后依次安排了 MEM/WB 前递验证段（生产者与消费者间隔两条 NOP，测试跨级前递）、混合前递验证段（一条 sub 指令同时依赖两条 addi 指令的结果，两个源寄存器均需前递），以及移位前递验证段（slli 依赖 addi 的结果）。40 轮测试中，每轮的 ALU 最终结果 x10 恒为 1024，周期数恒为 33，方差为零，证明前递数据通路的行为严格确定。

### 测试 7：控制流确定性（test7_branching.S）

本测试通过一个 4 级决策树来验证分支指令的控制流确定性。决策树以固定输入值 x10 = 42 驱动，在第 1 级用 bge 判断大小关系，第 2 级用 blt 判断范围，第 3 级用 andi 提取最低位后以 beq 判断奇偶性，第 4 级以 bge 进行最终分支。正确的执行路径为 R→L→L→L，终点寄存器 x12 被置为 0x309（即十进制 777）。其余 7 条错误路径的终点分别设置了互不相同的"陷阱值"，任何偏离正确路径的执行都会被最终比对检测。40 轮测试中，每轮均沿 R→L→L→L 路径执行，x12 恒为 777，周期数恒为 16，方差为零，证明在固定输入下控制流路径和执行时延完全确定。

### 测试 8：访存确定性（test8_memdep.S）

本测试通过四阶段访存序列来验证存储器读写操作的确定性。第一阶段向地址 `[0x100]` 至 `[0x114]` 依次写入 6 个不同数据字（0x111 至 0x666）；第二阶段以乱序方式读回这 6 个地址的值，逐一与写入值进行比对；第三阶段对 `[0x100]` 和 `[0x114]` 执行覆盖写入（0x999 和 0x888），随后读回以验证新值已完全替代旧值且无残留；第四阶段执行背靠背的 sw-lw 操作——对同一地址 `[0x120]` 先写入 0x777 再立即读取，验证写后即读的数据一致性。40 轮测试中，所有访存比对全部通过，周期数恒为 41，方差为零，证明 load/store 通路的行为严格确定。

### 测试 9：混合负载确定性（test9_mixedwork.S）

本测试通过在一个迭代内混合排列 ALU 运算、存储器访问、循环控制、函数调用和 CSR 寄存器读写五类操作，来验证处理器在复杂指令混合场景下的整体确定性。Block 1 包含 add、sub、slli 运算链，Block 2 执行三次 store 后三次 load 的交叉比对，Block 3 通过 10 次迭代的循环累加（addi + bnez）验证循环控制与时延，Block 4 使用 jal 和 jalr 完成一次子函数调用与返回，Block 5 两次读取 mcause CSR 并比对。40 轮测试中，全部 5 个计算块的中间结果和最终结果均与预期一致，周期数恒为 45，方差为零，证明处理器在混合指令负载下的行为严格确定。

测试 8 – 中断（interrupt）
本测试通过定时器中断直接验证中断延迟的确定性以及影子寄存器的保存/恢复正确性。测试程序首先将 x1–x5 初始化为已知值（0xA1–0xA5），配置 mtvec 指向 ISR 入口地址 0x90，使能定时器中断（MTIE）和全局中断（MIE），随后进入 40 轮迭代的主循环。每轮迭代中，主循环等待定时器中断触发——ISR 在进入后立即读取 mcycle 作为起始时间戳，随后故意将 x1 改写为 0xB1（破坏原值 0xA1），向地址 [0x200] 写入标志 1 通知主循环，再次读取 mcycle 计算 ISR 占用周期数并存入 [0x204]，最后执行 mret 返回。MRET 返回后，影子寄存器应将 x1 恢复为中断前的原值 0xA1；主循环读取 [0x204] 获得本轮 ISR 周期数并保存至内存，随后比对 x1 是否等于 0xA1——若不等则判定失败。该测试同时验证了三个关键特性：(1) 中断延迟的确定性——40 轮中断的 ISR 周期数应完全相等；(2) 影子寄存器的中断保存——ISR 内对 x1 的破坏不影响主循环上下文；(3) 影子寄存器的 MRET 恢复——返回后 x1 正确恢复为 0xA1。实测 40 轮中断的 ISR 周期数恒为 8，x1 每次均正确恢复为 0xA1，方差为零，证明中断响应时延和影子寄存器上下文保存/恢复行为严格确定。

## 二、English Version: Test Descriptions and Results

### Test 6: Forwarding Determinism (test6_forwarding.S)

This test validates the determinism of the pipeline forwarding paths by constructing a deep data-dependency chain consisting of ten consecutive addi instructions. In this chain, the source operand rs1 of each instruction is the destination register rd of its immediate predecessor, with no pipeline bubbles inserted between them. This forces the EX stage to retrieve forwarded data from the EX/MEM pipeline register on every cycle. Three additional forwarding scenarios are subsequently evaluated: a MEM/WB forwarding segment in which the producer and consumer are separated by two NOP instructions to exercise cross-stage forwarding; a dual-dependency segment in which a single sub instruction consumes results from two independent addi instructions, requiring simultaneous forwarding to both source operands; and a shift-forwarding segment in which an slli instruction depends on the output of a preceding addi. Across 40 iterations, the final ALU result in register x10 remains constant at 1024, and the cycle count per iteration remains constant at 33, yielding zero variance. These results confirm that the forwarding data paths behave in a strictly deterministic manner.

### Test 7: Control Flow Determinism (test7_branching.S)

This test validates the determinism of branch-induced control flow through a four-level decision tree driven by a fixed input value of x10 = 42. The first level employs bge to evaluate a magnitude comparison; the second level employs blt to evaluate a range condition; the third level extracts the least significant bit of x10 via andi and branches via beq to evaluate parity; the fourth and final level employs bge to perform the terminal branch. The correct execution path is R→L→L→L, culminating in register x12 being set to 0x309 (decimal 777). Each of the seven alternative paths sets x12 to a distinct trap value, ensuring that any deviation from the correct path is detected by the final comparison. Across 40 iterations, the execution consistently follows the R→L→L→L path, x12 remains constant at 777, and the cycle count per iteration remains constant at 16, yielding zero variance. These results confirm that, under fixed input conditions, both the control flow path and the execution latency are fully deterministic.

### Test 8: Memory Access Determinism (test8_memdep.S)

This test validates the determinism of memory read and write operations through a four-phase memory access sequence. In Phase 1, six distinct data words (0x111 through 0x666) are sequentially written to addresses [0x100] through [0x114]. In Phase 2, these six addresses are read back in a non-sequential order, and each loaded value is compared against its corresponding stored value. In Phase 3, overwrite writes of 0x999 and 0x888 are performed to addresses [0x100] and [0x114], respectively, followed by immediate read-back to verify that the new values have completely replaced the old values without residual corruption. In Phase 4, a back-to-back sw-lw pair targeting address [0x120] with value 0x777 is executed to verify write-after-read data consistency within the same memory location. Across 40 iterations, all memory comparisons pass, and the cycle count per iteration remains constant at 41, yielding zero variance. These results confirm that the load/store memory path behaves in a strictly deterministic manner.

### Test 9: Mixed-Workload Determinism (test9_mixedwork.S)

This test validates the holistic determinism of the processor under a mixed instruction workload by combining ALU operations, memory accesses, loop control, function calls, and CSR register reads within a single iteration. Block 1 consists of an arithmetic-logical chain involving add, sub, and slli operations. Block 2 performs three store operations followed by three load operations with cross-validation of the loaded values. Block 3 implements a ten-iteration accumulation loop using addi and bnez to verify control flow regularity and timing. Block 4 executes a subroutine call and return via jal and jalr to validate the function call mechanism. Block 5 reads the mcause CSR register twice and compares the two values to verify CSR access consistency. Across 40 iterations, all intermediate and final results from all five blocks remain consistent with expected values, and the cycle count per iteration remains constant at 45, yielding zero variance. These results confirm that the processor exhibits strictly deterministic behavior under a heterogeneous instruction mix representative of real application workloads.

 Test 8 — Interrupt. This test directly validates the determinism of interrupt latency and the correctness of
  shadow register save/restore through timer-triggered interrupts. The test program first initializes registers
  x1–x5 to known values (0xA1–0xA5), configures mtvec to point to the ISR entry at address 0x90, enables timer
  interrupts (MTIE) and global interrupts (MIE), and then enters a main loop of 40 iterations. In each
  iteration, the main loop waits for a timer interrupt to fire. Upon entry, the ISR immediately reads mcycle as
  the start timestamp, then deliberately overwrites x1 with 0xB1 (corrupting its original value 0xA1), writes a
  flag value of 1 to address [0x200] to notify the main loop, reads mcycle a second time to compute the ISR
  cycle count and stores it to [0x204], and finally executes an MRET to return. After MRET, the shadow registers
  shall restore x1 to its pre-interrupt value 0xA1; the main loop reads [0x204] to obtain the ISR cycle count
  of the current round and saves it to memory, then checks whether x1 equals 0xA1 — any mismatch is reported as
  a failure. This test simultaneously validates three critical properties: (1) interrupt latency determinism —
  the ISR cycle count shall be identical across all 40 interrupt rounds; (2) shadow register save on interrupt
  entry — the corruption of x1 inside the ISR does not affect the main-loop context; and (3) shadow register
  restore on MRET — x1 is correctly recovered to 0xA1 upon return. Across 40 interrupt rounds, the measured ISR
  cycle count remained constant at 8 cycles, x1 was correctly restored to 0xA1 in every round, and the variance
  was zero, confirming that both the interrupt response latency and the shadow register context save/restore
  behavior are strictly deterministic.
