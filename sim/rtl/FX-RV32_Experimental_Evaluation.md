IV. EXPERIMENTAL EVALUATION

A. Experimental Setup

We conducted comprehensive synthesis and FPGA verification of the FX‑RV32 core to evaluate its hardware overhead, timing determinism, interrupt latency, and GPIO control precision. All experiments were carried out on a uniform 55 nm process library and a Xilinx Kintex‑7 FPGA platform, and compared against multiple open-source RISC‑V cores.

1) Synthesis and ASIC Implementation. We used the SMIC 55LL HD RVT 55 nm CMOS standard cell library and Synopsys Design Compiler 2019.12 for synthesis. The synthesis clock was over-constrained to 3 ns to evaluate the maximum achievable performance of the core and the corresponding area under tight timing constraints, with the compile_ultra optimization strategy. Two configuration variants of FX‑RV32 were synthesized: a baseline version (RV32I only, shadow registers disabled) and a full version (RV32I + hardware shadow registers), in order to quantify the area cost of the shadow register module. For comparison, the open-source RISC‑V core OpenE902 (open-sourced by T-Head, RV32EMC, two-stage in-order pipeline, CLIC vectored interrupt controller, hardware multiplier/divider) was synthesized under the same library and constraints. In addition, we reproduced six configuration variants of Sophon [1] — baseline RV32I, with CLIC extension, and with EEI external interfaces scaled from 4 to 8/16/32 channels — to observe the scaling effects of different feature sets and EEI channel counts on area and timing. All designs used soc_top as the top-level module with external interfaces (EXT_ACCESS/EXT_INST/EXT_DATA) disabled, and the synthesis clock was over-constrained to 3 ns to evaluate peak performance. Area was obtained from cell_area and converted to NAND2 gate equivalents (1 GE ≈ 1.12 µm²); the critical path was obtained from report_timing. The synthesis results are summarized in Fig. 14.

(Insert Fig. 14 — Synthesis results: area [µm²] and critical path [ns] comparison of FX‑RV32, FX‑RV32+Shadow, Sophon (RV32I / +CLIC / +EEI4 / +EEI8 / +EEI16 / +EEI32), and OpenE902.)

2) FPGA Prototyping. FX‑RV32 was deployed as a soft core on a Xilinx Kintex‑7 XC7K325T‑2FFG900C FPGA (Genesys 2 board). FPGA synthesis used Xilinx Vivado 2022.1 with a target operating frequency of 200 MHz (to facilitate waveform observation using a logic analyzer and oscilloscope). The core logic (excluding data RAM) occupies approximately 2,056 LUTs and 1,913 flip-flops. Both the instruction ROM and data RAM were instantiated as Block RAMs (4 KB each).

3) Comparison Targets.

- OpenE902: A two-stage in-order pipeline RISC‑V core open-sourced by T-Head Semiconductor, targeting low-cost embedded applications. It supports the RV32EMC instruction set (RV32E base integer + hardware multiplier/divider + compressed instructions) and M/U privilege modes, and is equipped with a CLIC vectored interrupt controller. We used its default configuration (RV32EMC, CLIC vector mode enabled) and synthesized it under the same library for a fair area and timing comparison. Its interrupt latency (hardware vectoring mode) and GPIO control latency (APB-based) also serve as a reference baseline for evaluating the real-time performance of FX‑RV32.

- Sophon [1]: A single-cycle RV32I core featuring custom fGPIO instructions (single-cycle GPIO control) and an EEI external interface, with CLIC extension support. The literature reports interrupt latency under three processing modes: 3 cycles in hardware vectoring + inline handler mode (2 cycles to load the vector address from L1 data memory, 1 cycle for control transfer); 7 cycles in non-vectored mode with the custom snapreg instruction (snapreg performs single-cycle context save/restore); and 39 cycles in non-vectored mode with a C ABI handler (16 caller-saved registers are preserved sequentially via software store/load instructions). The baseline area at 55 nm is approximately 28.6 kGE. Sophon represents the most prominent recent work in low interrupt latency and deterministic response; its area/frequency data (across multiple configurations) and interrupt latency data are taken directly from the literature for systematic comparison with FX‑RV32 along three dimensions: interrupt latency, GPIO latency, and hardware overhead.

- PicoRV32: An open-source RV32I core designed by Clifford Wolf, targeting minimal area. It employs a multi-cycle state machine implementation (CPI > 1) without pipelining, caches, or branch prediction. Its interrupt mechanism is a non-standard fixed-entry scheme — there are no standard CSRs such as mtvec, mcause, or mepc, nor a vector table. Upon an interrupt trigger, the hardware jumps to the hard-coded address PROGADDR_IRQ (0x10), and the ISR manages context save/restore and interrupt source identification through custom instructions (getq/setq/retirq/maskirq). Since the multi-cycle state machine must complete the current instruction before responding to an interrupt, its interrupt latency exhibits significant variation — 3 cycles in the best case, and up to several tens of cycles in the worst case (depending on the type of the interrupted instruction; division/multiplication instructions require up to 40 cycles to complete). PicoRV32 serves as the area floor reference among lightweight RISC‑V cores; its inclusion in the comparison illustrates where FX‑RV32 sits in the area–real-time performance trade-off spectrum, achieving deterministic low-latency interrupt performance at a moderate area cost.

- Fast Interrupt (CLIC hardware stacking scheme): Mao et al. [5] proposed a hardware stacking interrupt architecture based on the RISC‑V CLIC specification. Upon interrupt response, the hardware automatically pushes 10 caller-saved registers onto an internal stack, eliminating the software save instruction sequence in the ISR. The reported interrupt latency is 13 clock cycles, of which 11 cycles are consumed by the hardware stacking operation. This scheme represents the performance level achievable by the design approach of "hardware-accelerated context saving without shadow registers," and serves to evaluate the latency advantage of full parallel shadow register saving over serial hardware stacking.

- CV32E40P [4]: An open-source four-stage in-order pipeline RISC‑V core (formerly RI5CY from the PULP platform), supporting the RV32IMFC instruction set and equipped with a custom vectored interrupt controller. Its interrupt latency is 33 cycles under the regular ABI, reduced to 24 cycles after EABI optimization — in both modes, the most time-consuming operation is software context save/restore executed instruction by instruction within the ISR. CV32E40P represents the performance upper bound of interrupt architectures without any hardware context acceleration, and its latency data inversely validates the effectiveness of hardware shadow registers in eliminating software save overhead.

4) Measurement Methodology.

- Area/Frequency: Post-synthesis cell area is reported by DC and converted to NAND2 gate equivalents (1 GE = 1.12 µm²); the critical path is obtained from report_timing.
- Determinism Tests: Test programs are compiled into RV32I machine code and loaded into the instruction ROM. The mcycle CSR is read via the JTAG debug interface. Each test is executed 40 times and the cycle counts are recorded.
- Interrupt Latency: A GPIO pin generates an external interrupt (rising-edge triggered), and the first instruction of the ISR asserts another GPIO output. A logic analyzer (Agilent 16822A) measures the number of clock cycles from the interrupt request signal to the ISR output transition, subtracting the instruction execution overhead within the ISR (1 cycle) to obtain the interrupt latency.
- GPIO Output Latency: The GPIO_OUT register is continuously written with 0xFFFFFFFF and 0x00000000 in a loop. An oscilloscope (Tektronix MDO3024) measures the delay from the execution of the store instruction to the pin-level transition.



B. Hardware Overhead and Frequency

1) Synthesis Results.

Fig. 14 summarizes the area and critical path of all synthesis configurations. The FX‑RV32 baseline version (RV32I only, shadow registers disabled) achieves a synthesis area of 27,191 µm² (approximately 24.3 kGE) at 55 nm, with a critical path of 4.89 ns, corresponding to a maximum frequency of 204 MHz. Among RISC‑V cores that feature pipelining or interrupt acceleration, this is the smallest area — even slightly lower than the Sophon RV32I baseline (32,116 µm², approximately 28.7 kGE). This demonstrates that the five-stage pipeline design is not inferior to single-cycle architectures in area efficiency: the additional flip-flop overhead introduced by pipeline registers (approximately 2 kGE) is offset by the larger combinational logic paths in single-cycle designs. The only area floor below this is the PicoRV32 baseline version (20,808 µm²), but it employs a non-pipelined multi-cycle state machine implementation with CPI ≈ 4, placing the two in different performance classes.

With hardware shadow registers enabled (full version), the area increases to 38,693 µm² (approximately 34.5 kGE), an increase of about 11,500 µm² (+42%). These 11,500 µm² represent the hardware cost of 31 32-bit shadow registers and their read/write control logic. Notably, the introduction of shadow registers has virtually no impact on the critical path (4.89 ns → 4.88 ns), because the shadow register save/restore operations do not lie on the critical path from the ALU to the data RAM address decoding — they constitute a parallel internal path within the register file. This means the shadow registers trade pure area for single-cycle context save capability without sacrificing frequency.

OpenE902, synthesized under the same library and constraints, occupies an area of 83,924 µm² (approximately 74.9 kGE), which is 3.1× the FX‑RV32 baseline and 2.2× the full version. Its critical path of 5.63 ns (maximum 178 MHz) is approximately 15% longer than that of FX‑RV32. The additional area of OpenE902 primarily stems from the RV32C compressed instruction decoder, the hardware multiplier/divider unit, and the more complex CLIC vectored interrupt controller — features that significantly increase hardware overhead while enhancing functionality.

2) Comparison with Sophon Configurations.

The six configuration variants of Sophon [1] illustrate the scaling effects of different feature sets on area and timing. The Sophon RV32I baseline occupies 32,116 µm² (approximately 28.7 kGE). Adding the CLIC extension increases the area to 34,115 µm² (+6.2%) while the critical path remains unchanged (5.04 ns), indicating that the CLIC logic is off the critical path. Adding a 4-channel EEI external interface further increases the area to 37,939 µm² (+18.1% relative to baseline), with the critical path slightly increasing to 5.10 ns.

The scaling of EEI channel count clearly reveals a linear area–interface bandwidth trade-off: from 4 to 32 channels, the area grows from 37,939 µm² to 70,737 µm² (+86.4%), while the critical path remains within the narrow range of 5.08–5.10 ns. This trend confirms that the EEI interface logic is not on the critical path — increasing the channel count merely replicates the interface datapath without affecting the combinational logic depth. At 32 channels, the Sophon area (70,737 µm²) approaches that of OpenE902 (83,924 µm²), yet its critical path remains significantly superior.

3) Area Comparison with PicoRV32.

The two configuration variants of PicoRV32 represent two typical operating points in area-prioritized design strategies. Its baseline version (base) occupies only 20,808 µm² (approximately 18.6 kGE) with a critical path of 4.90 ns — the smallest area among all 11 synthesis samples. This confirms its position as the area floor for RISC‑V soft cores, but at the cost of low instruction throughput due to the multi-cycle state machine (CPI ≈ 4) and non-deterministic interrupt response (latency varying from 3 to several tens of cycles).

PicoRV32's full version (full, with IRQ, QREGS, timer, and all interrupt features enabled) increases the area to 33,086 µm² (approximately 29.5 kGE, +59%), with the critical path slightly increasing to 4.95 ns. Notably, the FX‑RV32 baseline version (27,191 µm², 24.3 kGE) is 18% smaller than the PicoRV32 full version, yet delivers the instruction-level parallelism of a five-stage pipeline (CPI approaching 1, far superior to PicoRV32's CPI ≈ 4) and fully deterministic execution behavior. Even with shadow registers enabled (38,693 µm²), FX‑RV32 is only 17% larger than the PicoRV32 full version, while additionally providing single-cycle hardware context save capability — compressing the save/restore of 31 registers in the ISR from tens of software store/load instructions to 1 clock cycle, making the total latency from interrupt trigger to ISR context save completion far superior to PicoRV32. Combined with FX‑RV32's constant 2-cycle interrupt response latency (guaranteed by the interrupt pipeline architecture and independent of shadow registers), this stands in stark contrast to PicoRV32's interrupt latency variation from 3 to several tens of cycles.

In summary, the FX‑RV32 baseline version achieves the best critical path among all compared cores (4.88–4.89 ns) at an area of only 24.3 kGE. The 10.2 kGE area increment of the shadow registers (+42%) reduces the context save/restore overhead to 1 clock cycle at a pure area cost, without sacrificing frequency. The 2-cycle interrupt response latency of FX‑RV32 is independently guaranteed by the unconditional acceptance mechanism of the interrupt pipeline and hardware direct vector address computation — the two mechanisms work in concert, giving FX‑RV32 a comprehensive lead in end-to-end interrupt latency. In a direct comparison with the PicoRV32 full version, FX‑RV32 delivers significantly superior instruction throughput, interrupt response speed, and execution determinism at a comparable or even smaller area — precisely the most valuable competitive attributes in embedded real-time scenarios.



C. Deterministic Execution Verification

To rigorously verify the temporal repeatability of FX‑RV32, we selected four representative workloads and executed each independently 40 times, recording the mcycle difference.

Test 1 — Vector Addition (vvadd). Implements for (i = 0; i < 10; i++) C[i] = A[i] + B[i]. The loop is unrolled 10 times; each iteration comprises 10 loads (2 cycles each), 10 additions (1 cycle each), 10 stores (2 cycles each), and loop control instructions. The theoretical cycle count is 10 × (2 + 1 + 2) + branch overhead = 143 cycles. All 40 measured runs yielded 143 cycles, with a standard deviation of 0.

Test 2 — Recursive Fibonacci (fib(8)). A C recursive function int fib(int n), called with fib(8). The recursion depth is 8, involving function calls (JAL/JALR) and returns (RET), along with stack operations (addi sp, sp, -X, sw, lw). Since there is no branch predictor, all call paths are fixed. Each invocation measured 97 cycles, constant across 40 runs.

Test 3 — Bubble Sort. Sorts 10 32-bit integers in ascending order with worst-case input (reverse order). The sort requires 9 passes with a decreasing number of comparisons per pass. Since the processor applies a conditional branch penalty (1-cycle for ROM, 2-cycle for BRAM, per SYNC_INST_MEM parameter) (1 cycle for not-taken), the total cycle count is entirely determined by the instruction sequence. The measured result was 1,081 cycles across 40 runs, with zero jitter.

Test 4 — Linear Feedback Shift Register (LFSR). A 32-bit LFSR is implemented using the CRC-32 polynomial (0xEDB88320), executing 256 iterations. Each iteration extracts the least significant bit, right-shifts, and conditionally XORs based on the extracted bit. The loop contains no branches (using conditional moves or bitwise operations), and all instruction latencies are fixed. The measured result was 312 cycles across 40 runs, constant.


Test 5 — Forwarding. A deep data dependency chain consisting of 10 consecutive addi instructions is constructed to verify the determinism of the pipeline forwarding paths. In the chain, the source operand rs1 of each instruction is the destination register rd of the preceding instruction, with no bubbles between adjacent instructions, forcing the EX stage to obtain forwarded data from the EX/MEM pipeline register in every cycle. This is followed by a MEM/WB forwarding verification segment (producer and consumer separated by two NOPs, testing cross-stage forwarding), a mixed forwarding verification segment (a sub instruction simultaneously dependent on the results of two addi instructions, requiring both source registers to be forwarded), and a shift forwarding verification segment (slli dependent on the result of addi). Across 40 runs, the final ALU result in x10 was consistently 1,024, the cycle count was consistently 33, and the variance was zero, confirming that the forwarding datapath behavior is strictly deterministic.

Test 6 — Branching. This test verifies the control flow determinism of branch instructions via a four-level decision tree. The decision tree is driven by a fixed input value x10 = 42: Level 1 uses bge to evaluate a magnitude relationship; Level 2 uses blt to evaluate a range; Level 3 uses andi to extract the least significant bit and then beq to evaluate parity; Level 4 uses bge for the final branch. The correct execution path is R → L → L → L, with the destination register x12 set to 0x309 (decimal 777). The remaining seven incorrect paths set mutually distinct "trap values" in their respective destination registers; any deviation from the correct path is detected by the final comparison. Across 40 runs, every run followed the R → L → L → L path, x12 was consistently 777, the cycle count was consistently 16, and the variance was zero, confirming that both the control flow path and execution latency are fully deterministic under fixed inputs.

Test 7 — Memory Dependence (memdep). This test verifies the determinism of memory read/write operations through a four-phase memory access sequence. Phase 1 sequentially writes six distinct data words (0x111 through 0x666) to addresses [0x100] through [0x114]. Phase 2 reads back the values from these six addresses in a non-sequential order and compares each against the written value. Phase 3 performs overwriting writes (0x999 and 0x888) to [0x100] and [0x114], then reads back to verify that the new values have completely replaced the old ones with no residual. Phase 4 executes a back-to-back sw-lw operation — writing 0x777 to address [0x120] and immediately reading it back — to verify write-after-read data consistency. Across 40 runs, all memory access comparisons passed, the cycle count was consistently 41, and the variance was zero, confirming that the load/store datapath behavior is strictly deterministic.

Test 8 — Interrupt. This test directly validates interrupt latency determinism and shadow register save/restore correctness through timer-triggered interrupts. The test program first initializes registers x1–x5 to known values (0xA1–0xA5), configures mtvec to point to the ISR entry at address 0x90, enables timer interrupts (MTIE) and global interrupts (MIE), and then enters a main loop of 40 iterations. In each iteration, the main loop waits for a timer interrupt to fire. Upon entry, the ISR immediately reads mcycle as the start timestamp, then deliberately overwrites x1 with 0xB1 (corrupting its original value 0xA1), writes a flag value of 1 to address [0x200] to notify the main loop, reads mcycle a second time to compute the ISR cycle count and stores it to [0x204], and finally executes an mret to return. After mret, the shadow registers shall restore x1 to its pre-interrupt value 0xA1; the main loop reads [0x204] to obtain the ISR cycle count of the current round and saves it to memory, then checks whether x1 equals 0xA1 — any mismatch is reported as a failure. This test simultaneously validates three critical properties: (1) interrupt latency determinism — the ISR cycle count shall be identical across all 40 interrupt rounds; (2) shadow register save on interrupt entry — the corruption of x1 inside the ISR does not affect the main-loop context; and (3) shadow register restore on mret — x1 is correctly recovered to 0xA1 upon return. Across 40 interrupt rounds, the measured ISR cycle count remained constant at 8 cycles, x1 was correctly restored to 0xA1 in every round, and the variance was zero, confirming that both the interrupt response latency and the shadow register context save/restore behavior are strictly deterministic.

D. Interrupt Latency

To evaluate the interrupt response behavior of FX‑RV32, we measured the number of clock cycles from the acceptance of an interrupt request by the core to the fetch of the first instruction of the interrupt service routine (ISR). This measurement covers only the hardware path of interrupt response — including interrupt arbitration, vector address computation, pipeline flush, and PC redirection — and excludes the context save/restore operations inside the ISR; in FX‑RV32, the latter is performed in parallel by the hardware shadow registers within a single clock cycle and constitutes a time component independent of the interrupt response latency. The test used a GPIO external interrupt (interrupt ID 11), with mtvec configured in vectored mode (MODE = 01) and the base address set to 0x200, yielding a vector table entry address of 0x200 + 11 × 4 = 0x22C. The first instruction of the ISR was a nop (to facilitate waveform observation), and the ISR concluded with an mret. A logic analyzer (Agilent 16822A) simultaneously captured the interrupt request signal intr_external_i, the interrupt acceptance flag interrupt_o, the shadow register control signal shadow_save, and the program counter pc bus, measuring the number of clock cycles from the rising edge of intr_external_i to the entry of the first ISR instruction into the fetch stage. For comparison, we measured the interrupt latency of OpenE902 in CLIC hardware vectoring mode on the same FPGA platform (identical interrupt source and triggering method). The interrupt latency data for Sophon [1], Fast Interrupt (CLIC hardware stacking scheme) [5], and CV32E40P [4] were taken from the respective literature; the interrupt latency of PicoRV32 was obtained by injecting GPIO interrupts and observing waveforms in its Verilator simulation environment. The experimental results are summarized in Fig. 16.

(Insert Fig. 16 — Interrupt latency comparison among RISC‑V CPUs: Sophon (C ABI 39 cycles / snapreg 7 cycles / hardware vectoring 3 cycles), OpenE902 (hardware vectoring 9 cycles), Fast Interrupt (hardware stacking 13 cycles), CV32E40P (EABI 24 cycles / regular ABI 33 cycles), PicoRV32 (non-vectored custom retirq, fastest 3 cycles), FX‑RV32 (hardware vectoring 2 cycles).)

The experimental results show that the interrupt latency of FX‑RV32 is 2 clock cycles. The timing is as follows: in the first clock cycle, the interrupt controller completes priority arbitration and asserts interrupt_o, and the interrupt pipeline unconditionally accepts the interrupt request at any execution stage of any instruction — similar to Sophon but further eliminating the blocking conditions imposed by EX-stage branch/jump instructions and MEM-stage load instructions on interrupt acceptance (see Section III for the interrupt pipeline architecture), and the shadow register save signal (shadow_save) is asserted to 1 immediately thereafter. In the second clock cycle, the hardware combinational logic directly computes the vector address BASE + cause × 4 and redirects the PC to 0x22C; upon detecting the save signal, the shadow registers complete the parallel save of x1–x31 within the same cycle while concurrently completing the original write-back operation. The first ISR instruction (nop) enters the IF stage at the end of the second cycle. Across 1,000 consecutive triggers, the latency remained constant at 2 cycles with no variation whatsoever — this determinism is guaranteed by the unconditional acceptance mechanism of the interrupt pipeline and is entirely independent of the type of the interrupted instruction (ALU/load/store/branch/jump) and the pipeline state.

Sophon [1] achieves an interrupt latency of 3 cycles in hardware vectoring mode, the closest to FX‑RV32 among all compared cores. Its 3-cycle composition is: 2 cycles to load the vector table address from L1 data memory, and 1 cycle to transfer control to the interrupt handler. The key reason FX‑RV32 is able to further compress the latency to 2 cycles lies in the different method of vector address computation — the interrupt controller of FX‑RV32 directly computes mtvec + cause × 4 using pure combinational logic, requiring no memory access to load the vector table, thereby merging address generation and PC redirection into a single cycle. In non-vectored mode, the interrupt latency of Sophon increases substantially: 39 cycles when using a C ABI handler — time mainly consumed by sequentially saving 16 caller-saved registers using store instructions; even with the custom snapreg instruction compressing context save to a single cycle, the total latency remains 7 cycles (1 cycle for snapshot save + 6 cycles for the remaining hardware path).

OpenE902 exhibits an interrupt latency of 9 clock cycles in CLIC hardware vectoring mode. Its two-stage pipeline architecture requires a complete pipeline flush operation upon interrupt response, and the CLIC vector table loading latency exceeds that of Sophon's single-cycle design, ultimately resulting in an interrupt latency approximately 3× that of Sophon's hardware vectoring mode and 4.5× that of FX‑RV32. Fast Interrupt (CLIC hardware stacking scheme) [5] has an interrupt latency of 13 clock cycles, of which 11 cycles are consumed by the hardware serial stacking operation to save 10 caller-saved registers — a scheme that replaces the software store instruction sequence with a hardware state machine operating in serial fashion. Although this avoids instruction fetch/decode overhead, its serial nature prevents it from competing with the fully parallel save capability of shadow registers (31 registers saved in 1 cycle). CV32E40P [4] exhibits an interrupt latency of 33 cycles under the regular ABI, reduced to 24 cycles after EABI optimization — in both modes, the most time-consuming operation is the software context save/restore executed instruction by instruction within the ISR. CV32E40P represents the performance upper bound of interrupt architectures without any hardware context acceleration, and its latency data inversely validates the effectiveness of hardware shadow registers in eliminating software save overhead.

The interrupt latency of PicoRV32 is 3 cycles in the best case, but this figure must be interpreted with care in light of its architectural characteristics. First, PicoRV32 employs a non-standard interrupt mechanism — there are no standard RISC‑V CSRs such as mtvec, mcause, or mepc, nor a vector interrupt table. Upon an interrupt trigger, the hardware forces a jump to the hard-coded fixed entry address PROGADDR_IRQ (0x10). A custom getq instruction is typically placed at this entry point to read interrupt source information, after which the ISR sequentially polls the status registers of each peripheral via software to determine the interrupt source and dispatch to the corresponding handler routine. Consequently, its 3 cycles only cover the minimal path of "hardware detects interrupt → waits for current instruction to complete → jumps to 0x10," and do not include the overhead of vector dispatching (no vector table to query) or context saving (requiring subsequent getq/setq instructions performed manually). Second, the multi-cycle state machine architecture of PicoRV32 introduces significant non-determinism in its interrupt latency: when the interrupted instruction is a simple ALU instruction, the response can occur within 3 cycles; however, when the CPU is executing a multi-cycle instruction such as division (up to 40 cycles) or multiplication (up to 8 cycles), the state machine must wait for the current instruction to fully complete before transitioning to the interrupt handling state, resulting in latency variation reaching several tens of cycles. In contrast, the 2-cycle latency of FX‑RV32 is strictly deterministic — the interrupt pipeline unconditionally accepts interrupts at any stage of any instruction, exhibiting no variation due to instruction type. Furthermore, FX‑RV32 supports the standard RISC‑V vectored interrupt mode, with the hardware automatically indexing the vector table by interrupt ID and jumping to the corresponding handler routine, requiring no software interrupt source querying within the ISR — this hardware vector dispatching capability is also not accounted for in PicoRV32's best-case 3-cycle latency.

In summary, among these RISC‑V cores, FX‑RV32 achieves the lowest interrupt response latency of 2 clock cycles — a 33% reduction compared to the previously best Sophon hardware vectoring mode (3 cycles). More critically, this latency is strictly deterministic, independent of the type of the interrupted instruction or the pipeline state. Combined with the single-cycle full register file context save capability provided by the hardware shadow registers, the end-to-end latency from interrupt trigger to the ISR beginning to execute actual processing logic is merely 3 clock cycles (2 cycles response + 1 cycle context save) — a figure that remains superior to the interrupt response latency alone (excluding context save) of all compared cores. This fully demonstrates the advantage of the synergistic design of the interrupt pipeline and shadow register mechanisms.



E. Chapter Summary

This chapter has systematically evaluated FX‑RV32 across three dimensions: hardware overhead, execution determinism, and interrupt response latency. In terms of hardware overhead, the FX‑RV32 baseline version (RV32I only) occupies a synthesis area of merely 24.3 kGE at 55 nm — the smallest among RISC‑V cores featuring pipelining or interrupt acceleration — with a critical path of 4.89 ns (204 MHz), the best among all 11 synthesis samples. With shadow registers enabled, the area increases to 34.5 kGE (+42%), trading pure area for single-cycle full register file context save capability without affecting the critical path. In terms of execution determinism, eight test programs covering ALU operations, memory access, branching, forwarding, and interrupt mixed scenarios all exhibit zero cycle count standard deviation across 40 repeated runs, verifying that the five-stage pipeline microarchitecture without caches or branch prediction possesses perfect temporal repeatability. In terms of interrupt response, FX‑RV32 achieves a deterministic interrupt latency of 2 clock cycles — the lowest publicly reported to date — by virtue of the unconditional acceptance mechanism of the interrupt pipeline and hardware direct vector address computation. With shadow registers, the full pipeline from interrupt trigger to ISR context save completion requires only 3 clock cycles, still surpassing the interrupt response latency alone of all compared cores. Furthermore, FX‑RV32 can toggle a pin level within 1 clock cycle via a standard sw instruction to an MMIO GPIO register (Fig. XX), matching the response speed of Sophon's custom fGPIO instruction without requiring any custom instruction extensions. Taken together, FX‑RV32, with a lightweight area of 24.3–34.5 kGE, matches or exceeds the best-in-class RISC‑V cores in the three key metrics of interrupt real-time performance, execution determinism, and I/O response speed, validating the effectiveness of the synergistic interrupt pipeline and shadow register design.
