## 1. Store 指令无法写入 data_ram

### 现象

所有 UVM 测试中，`sw` 指令均无法完成数据写入：
- Monitor 检测到 **0 次 STORE 事件**（条件为 `bus_re && bus_we`）
- Load 指令从目标地址读回数据均为 `0xxxxxxxxx`（未初始化）
- 最小化测试（只有一条 `sw x10, 0x100(x0)`）同样复现

### 根因

**文件**: `core/mem/mem_top.v:53`

```verilog
// 原始代码
assign bus_re_o = mem_re_i;   // ← SW 指令 mem_re_i=0, 所以 bus_re=0
assign bus_we_o = mem_we_i;   //   SW 指令 mem_we_i=1, bus_we=1
```

CPU 的 `mem_top` 将 `bus_re`（总线请求）和 `bus_we`（写使能）分开驱动：
- **LOAD**: `bus_re=1, bus_we=0`
- **STORE**: `bus_re=0, bus_we=1`（`mem_re_i` 为 0，`bus_re_o` 被拉低）

而 UVM testbench 中 data_ram 的写入条件是 **`vif.bus_re && vif.bus_we`**——两个信号必须同时为高：

```verilog
// uvm_tb_top.sv
always @(posedge clk) begin
    if (vif.bus_re && vif.bus_we) begin  // ← 要求两个都是 1
        data_ram[addr] <= vif.bus_wdata;
    end
end
```

对于 STORE 指令，`bus_re=0, bus_we=1` → 条件不成立 → 写入永不触发。

### 修复

```verilog
// 修复后: bus_re 对读和写都拉高
assign bus_re_o = mem_re_i || mem_we_i;
assign bus_we_o = mem_we_i;
```

`bus_re` 现在既是读请求也是写请求信号。对 STORE 指令，`bus_re=1, bus_we=1`，满足 testbench 的写入条件。

**影响范围**: 所有依赖 store 的测试，包括 hazard、interrupt、alu 测试。

## 2. ctrl 模块端口未连接警告

### 现象

Modelsim 编译时产生多个警告：

```
Too few port connections for 'u_ctrl'. Expected 14, found 9.
Missing connection for port 'mem_write_o'.
Missing connection for port 'mem_read_o'.
Missing connection for port 'reg_write_o'.
Missing connection for port 'mem_to_reg_o'.
Missing connection for port 'alu_src_o'.
```

### 根因

**文件**: `core/id/id_top.v:134-147`, `core/id/ctrl.v`

`ctrl` 模块和 `decoder` 模块存在大量功能重叠。`id_top.v` 中 `ctrl` 的 11 个输出端口只连接了 `branch_o` 和 `jump_o` 两个，其余 9 个端口从未使用（因为 `decoder` 提供了相同信号）。

`ctrl.v` 原始版本有 78 行，生成了 11 个控制信号，但实际只用到了分支和跳转判断。

### 修复

将 `ctrl.v` 精简为只包含实际需要的端口（从 78 行减少到 37 行）：

```verilog
// 修复后: ctrl 只负责分支和跳转判断
module ctrl (
    input  wire [6:0]  opcode_i,
    input  wire [2:0]  funct3_i,
    input  wire [6:0]  funct7_i,
    input  wire [31:0] instr_i,
    output wire        branch_o,
    output wire        jump_o
);
```

同时清理了 `id_top.v` 中 `ctrl` 实例化的 3 个空端口连接（`csr_inst_o`, `csr_write_o`, `mret_o`）。

**编译警告从 15 条减少到 9 条**（剩余警告为 RTL 内部未连接端口，属设计预留，不影响功能）。

## 3. data_ram 未初始化导致 tohost 检测误判

### 现象

`cpu_test_hazard` 测试中，Test1 的检查结果总是 `got=0xxxxxxxxx`，即使 store 修复后数据实际上已正确写入。

### 根因

**文件**: `uvm/uvm_tb_top.sv`

`data_ram` 声明为 `logic [31:0] data_ram [0:16383]`，在 SystemVerilog 仿真中 `logic` 类型默认为 `X`（未知状态）。

UVM 测试类 `cpu_test_hazard` 的检查逻辑：

```verilog
wait($root.uvm_tb_top.data_ram[63] !== 32'h0);  // 等待 tohost 变化
r1 = $root.uvm_tb_top.data_ram[63];              // 读取结果
```

仿真开始时 `data_ram[63] = 32'hxxxxxxxx`。`X !== 0` 在 Verilog 中立即成立，所以测试**在第一次 store 之前**就读到了 X，导致：
- Test1 读到 `X` → 与预期 `0xDEADBEF0` 比较 → [FAIL]
- Test2 读到 Test1 的真实结果 → 与预期 `0xCAFEBBBE` 比较 → [FAIL]
- Test3 读到 Test2 的真实结果 → 与预期 `0xDEADBEEF` 比较 → [FAIL]

所有结果向下偏移一位，三个测试全部错位失败。

### 修复

在 `uvm_tb_top.sv` 的 `initial` 块中，于复位之前添加 data_ram 初始化：

```verilog
// 初始化 RAM: 全部填 0, 避免 X 值干扰测试判断
for (int i = 0; i < 16384; i++)
    data_ram[i] = 32'h00000000;
```

`inst_rom` 已有类似初始化（填充 NOP），`data_ram` 遗漏了。

## 4. 影子寄存器未使能

### 现象

中断测试 `cpu_test_interrupt` 失败：ISR 破坏 x1-x5 后 MRET 返回，寄存器未恢复。

### 根因

**文件**: `core/id/id_top.v:18`, `core/interrupt/interrupt_pipeline.v:22`

两个模块的 `SHADOW_EN` 参数均默认为 `0`（禁用状态）：

```verilog
// id_top.v
parameter SHADOW_EN = 0    // 影子寄存器使能: 1=开启, 0=关闭

// interrupt_pipeline.v
parameter SHADOW_EN = 0    // 影子寄存器使能: 1=开启, 0=关闭
```

当 `SHADOW_EN=0` 时，中断进入不保存 GPR（通用寄存器），MRET 不恢复 GPR。ISR 对寄存器的修改在返回后仍然存在。

### 修复

```verilog
// id_top.v
parameter SHADOW_EN = 1    // 已修复

// interrupt_pipeline.v
parameter SHADOW_EN = 1    // 已修复
```

注意：两个参数必须同时设置，否则会导致部分操作（只保存不恢复，或只恢复不保存）。

CLAUDE.md 中 `doc/interrupt/shadow_en_config.md` 已有详细说明。

## 6. 测试程序问题

### 6.1 `lui` 立即数格式

**文件**: `uvm/load_use_test.s`

汇编器 `riscv_asm7.py` 的 `encode_u_type` 在编码 `lui` 时将输入值右移 12 位再左移 12 位，期望用户传入完整的 32 位值：

```asm
# 错误: 0xDEADB 只有 20 位, 被 >>12 再 <<12 后丢失高 8 位
lui x10, 0xDEADB        → x10 = 0x000DE000  ✗

# 正确: 传入完整的 32 位值
lui x10, 0xDEADB000     → x10 = 0xDEADB000  ✓

# 更推荐: 使用 li 伪指令
li  x10, 0xDEADBEEF     → x10 = 0xDEADBEEF  ✓
```

### 6.2 `ori` 符号扩展

**文件**: `uvm/load_use_test.s`

RISC-V 的 I-type 指令对 12 位立即数做**符号扩展**。当 bit11=1 时，立即数被视为负数：

```asm
# 错误: 0xEEF 的 bit11=1, 符号扩展为 0xFFFFFEEF = -273
lui x10, 0xDEADB        # → x10 = 0x000DE000
ori x10, x10, 0xEEF     # → x10 = 0x000DE000 | 0xFFFFFEEF = 0xFFFFFEEF ✗

# 正确: 用 li 伪指令, 自动处理符号扩展
li  x10, 0xDEADBEEF     # → x10 = 0xDEADBEEF  ✓
```

`li` 伪指令展开为 `lui(top+1) + addi(负偏移)`，正确补偿了符号扩展效应。

### 6.3 ISR 与影子寄存器的通信机制

**文件**: `uvm/intr_test.s`

原始测试程序使用寄存器 `x10` 作为 ISR→主循环的标志：

```asm
# 主循环
addi x10, x0, 0        # x10 = 0
wait_intr:
    bne x10, x0, check_result  # 等 x10 != 0

# ISR
addi x10, x0, 1        # 通知主循环
mret                   # ← 影子寄存器恢复 x10=0！
```

影子寄存器在 MRET 时恢复**所有** GPR（包括 x10），ISR 对 x10 的修改被撤销。修复：ISR 改用内存 `[0x200]` 作为中断标志（内存不受影子寄存器影响）。

### 6.4 `la` 伪指令展开不完整

**文件**: `uvm/intr_test.s`

`riscv_asm7.py` 的 `la` 只生成 `auipc rd, 0`，缺少 `addi` 偏移部分，导致 mtvec 指向错误地址。修复：手动用 `lui + addi` 硬编码 ISR 地址。

## 8. 残留问题

### 8.1 srli 指令疑似译码错误

**现象**: `srli x12, x10, 2` 在 x10=0xFFFFFFF0 时返回 0xFFFFFFF2 而非 0x3FFFFFFC，结果与 `srai`（算术右移）一致。

**可能原因**:
- 译码器 `decoder.v` 中 `alu_op_o` 对 `srli` 和 `srai` 的区分不正确
- 或 ALU 内部移位逻辑未正确处理 `funct7` 的 bit5

**复现**: 运行 `cpu_test_alu`

### 8.2 `la` 伪指令不完整

`riscv_asm7.py` 中 `la rd, symbol` 只展开为 `auipc rd, 0`，缺少 `addi rd, rd, offset`。用户表示该脚本有另外的用处暂不修改，当前通过手动硬编码地址绕过。

### 8.3 RTL 内部未连接端口

以下端口在 `core_top.v` 中被有意或无意地留空，产生 vopt 警告但不阻塞仿真：

| 模块 | 未连接端口 | 数量 |
|------|-----------|------|
| `ifu_top` | `instr` | 1 |
| `mem_top` | `mem_exception_o` | 1 |
| `id_top/ctrl` | 多个控制输出（已修复） | 5 |
