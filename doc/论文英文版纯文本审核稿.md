**Abstract:** 
Embedded real-time systems demand processors that guarantee predictable execution timing, ultra low interrupt latency, and minimal hardware area.
Mainstream RISC-V cores, while offering flexibility, often rely on caches, branch predictors, and deep pipelines, leading to timing anomalies and long interrupt response times.
This paper presents FX-RV32, a deterministic, low latency, lightweight RISC-V processor specifically designed for hard real-time control applications.
Its baseline core implements only the RV32I base integer instruction set, employs a five-stage in-order pipeline without branch prediction, and uses a tightly coupled, cache-less Harvard memory architecture.
Synthesized in 55 nm CMOS, the baseline core occupies only 24.9 kGE.
On this basis, we further introduce a shadow register extension that, upon an interrupt, hardware parallel saves all general-purpose registers, compressing the context save time to a single cycle.
FX-RV32 achieves a fixed interrupt latency of 2 clock cycles and a GPIO output latency of 1 cycle.
All instructions have fixed or statically known latencies, ensuring time-repeatable execution.
FPGA‑based deterministic validation using eight benchmark programs (covering ALU operations, memory accesses, branches, forwarding, and interrupt scenarios) showed zero execution time jitter across 40 runs.
Compared to state-of-the-art real-time RISC-V cores such as Sophon (3 cycle interrupt, 28.6 kGE) and CLIC-based designs (13 cycle interrupt), FX-RV32 offers the lowest interrupt latency and highly competitive hardware efficiency, making it an ideal choice for latency-sensitive, safety-critical, and resource-constrained embedded systems.

**Keywords:** 
RISC-V, deterministic processor, low interrupt latency, shadow register, cache-less Harvard architecture, time repeatability, hard real-time.

## Introduction

{ith} the rapid development of the Internet of Things, industrial automation,
automotive electronics, and robotics, embedded systems demand increasingly
stringent real-time performance, low power consumption, and small area.
RISC-V[ref7], as an open and modular instruction set architecture, has gained
widespread attention in both academia and industry in recent years. In hard
real-time systems, tasks must complete within specified deadlines; otherwise,
equipment damage, control failure, or even loss of life may occur. Interrupt
latency – the time from when an interrupt request is accepted by the core to
when the first instruction of the interrupt handler is executed – and GPIO
control latency directly determine the system's reaction speed to external
events. Furthermore, many control applications (e.g., motor drives, sensor
sampling) require strictly identical execution times under the same input, i.e.,
time repeatability, otherwise control jitter or sampling errors arise.

However, mainstream RISC-V processors, in pursuit of average performance,
commonly employ deep pipelines, branch prediction, out-of-order execution, and
cache hierarchies. While these techniques improve average execution efficiency,
they cause serious problems for hard real-time systems: branch mispredictions
introduce uncertain penalty cycles, cache hits/misses cause memory access time
jitter, and out-of-order execution makes worst-case execution time (WCET)
analysis extremely difficult. For example, the high-performance out-of-order
core BOOM[ref24] often exhibits interrupt latency exceeding 50 cycles, varying
with pipeline state; even some lightweight in order cores (such as the open
source PicoRV32 core[ref29]) suffer from interrupt latencies ranging from as low
as 7 cycles to as many as several tens of cycles due to the lack of hardware
interrupt acceleration.

Control latency, particularly GPIO latency and interrupt latency, is a key
metric for embedded real-time system performance. External devices connect to
the processor via GPIO pins and asynchronously send interrupt requests.
Traditional GPIO controllers are attached via on-chip buses (e.g., APB or AHB).
The CPU must execute a store instruction, which undergoes bus arbitration,
address decoding, register write, etc., before the pin state changes. For
instance, the APB GPIO controller in the OpenE902 processor[ref30] takes three
clock cycles from the store instruction to the actual pin toggle, and bus
contention may introduce additional delay. To reduce GPIO latency, TI's PRU
solution directly couples GPIO to the core via dedicated registers R30/R31,
achieving 5 ns toggling speed, but the dedicated registers reduce the number of
general-purpose registers and the design is closed source IP. GPIOCP[ref10]
designs a programmable I/O coprocessor that pre-loads command sequences to
achieve cycle-accurate GPIO control, but at the cost of additional hardware
resources. Sophon[ref1] uses a custom fGPIO instruction that encodes GPIO
operations directly into instructions, completing pin control in one cycle
through its EEI interface, but this requires ISA extension and increases
software porting complexity. Zhao et al.[ref11] improved the GPIOCP approach and
proposed two scheduling methods to further enhance system-level timing accuracy.

Interrupt latency is equally critical. In a standard RISC-V system, external
interrupts are first arbitrated by the Platform Level Interrupt Controller
(PLIC)[ref17]; the CPU must read the mcause register via software to identify
the interrupt source and then jump to the corresponding handler. Worse, upon
entering an interrupt, software must use store instructions to save the
interrupted program's general-purpose registers one by one, and restore them
with load instructions upon exit – a process that consumes dozens of
cycles[ref6]. To accelerate interrupt response, RISC-V introduced the Core Local
Interrupt Controller (CLIC) extension[ref18], supporting hardware vectoring
(direct calculation of the entry address from the interrupt ID) and interrupt
level preemption. Based on CLIC, Mao et al.[ref5] designed a hardware stacking
mechanism that automatically pushes 10 caller-save registers onto the memory
stack upon interrupt entry, reducing interrupt latency to 13 cycles. Balas and
Benini[ref6] reduced the number of registers to be saved by adopting the
Embedded ABI (EABI), lowering CV32E40P's interrupt latency to 24 cycles.
Sophon[ref1] combines CLIC with a custom snapreg instruction, achieving 3-cycle
interrupt latency in hardware vectoring mode, but snapreg still requires
software invocation; in non-vectoring mode the latency remains 7 cycles. In
summary, existing solutions have not reached a 2-cycle interrupt latency, and
most rely on custom instructions or additional memory stack operations.

As early as 2009, Edwards et al. emphasized the importance of repeatable timing
for embedded systems[ref19]. The academic community has long been designing
time-predictable or time-repeatable processors. Patmos[ref8] uses a fully
predictable instruction set, a method cache[ref9], and static scheduling;
SIC[ref3] rigorously proves timing predictability by eliminating dependencies
between consecutive instructions. However, these target WCET analysis and do not
achieve cycle-exact repeatable execution. Precision Timed ARM (PTARM)[ref2] uses
a thread-interleaved pipeline (four threads interleaved in a five-stage
pipeline) to achieve repeatable timing, building on the earlier PRET (Precision
Timed) machine concept[ref20]. Nevertheless, PTARM has low single-thread
performance and requires at least four active threads to avoid pipeline stalls.
FlexPRET[ref12] extends PTARM to support mixed criticality but retains the
thread-interleaved nature. Texas Instruments' PRU, using a single-cycle
architecture and dedicated registers, achieves low-latency GPIO and
deterministic execution, but it is a closed source commercial chip. Recently,
Sophon[ref1] proposed a single-cycle RV32I core that implements 1-cycle GPIO and
3-cycle interrupt (vectoring mode) using custom fGPIO and snapreg instructions,
while maintaining time repeatability, with a minimum area of 28.6 kGE. Although
Sophon is the closest existing work to our goal, its interrupt latency is still
not 2 cycles and it requires custom instruction extensions, sacrificing ISA
simplicity to some extent.

The RISC-V ecosystem offers many configurable cores — Rocket Chip[ref13],
PULP[ref14], Snitch[ref15] — and ISA extensions for custom
instructions[ref21,ref22], stream processing[ref23], and vector
acceleration[ref25]. However, none of these are specifically optimized for hard
real-time interrupt latency and determinism.

To address the above issues, we design a novel RISC-V processor named FX-RV32.
The processor uses a minimalist RV32I base, adopts a five-stage in-order
pipeline, completely removes branch predictors and caches, and employs a Harvard
architecture with tightly coupled instruction ROM and data RAM. Its baseline
core occupies only 24.9 kGE in 55 nm CMOS. To further shorten the general-
purpose register save time during interrupt response, we introduce a shadow
register extension that, upon an interrupt, hardware parallel saves all general-
purpose registers (x1–x31) and tightly couples with the interrupt pipeline. The
hardware automatically saves the machine mode critical state (mepc, mcause,
mstatus) and completes the vector jump in the next cycle, achieving a fixed
interrupt latency of 2 clock cycles. GPIO uses memory-mapped registers; the pin
updates on the next clock edge after a write, giving an output latency of 1
cycle. All instructions (except loads/stores which take 2 cycles, and branches
which take 2 or 3 cycles) have fixed latencies, making program execution fully
time-repeatable. We run eight deterministic tests on FPGA, each executed 40
times, showing zero jitter in cycle counts; measured interrupt latency is 2
cycles, and the maximum GPIO toggling frequency is half the core frequency.
Compared to Sophon, FX-RV32 reduces interrupt latency by 33% while maintaining
the standard RV32I ISA without any custom extensions.

The remainder of this paper is organized as follows. Section II reviews related
work on time-predictable/repeatable architectures, control latency
optimizations, and hardware acceleration techniques (shadow registers, interrupt
pipeline). Section III details the microarchitecture of FX-RV32, including the
five-stage pipeline, Harvard memory system, branch handling, shadow register
based interrupt mechanism, GPIO, and peripheral interfaces. Section IV presents
the experimental evaluation (hardware overhead, deterministic validation,
interrupt/GPIO latency measurements). Section V concludes the paper and outlines
future work.

## Related Work

### Time Predictable and Time Repeatable Architectures

Hard real-time systems require tight timing bounds; worst-case execution time
(WCET) analysis is a traditional method[ref7]. However, branch prediction,
speculative execution, and multi-level caches cause timing anomalies, where a
locally "better" situation can lead to a globally longer execution time[ref3].
Several time-predictable architectures have been proposed to address this.
Patmos[ref8] uses a predictable instruction set and method cache[ref9] with
compiler-scheduled dual issue, but still suffers from cache misses and is not
cycle-repeatable. SIC[ref3] enforces strictly in-order memory accesses to
eliminate timing anomalies; MINOTAuR[ref4] applies SIC principles on Ariane with
limited speculation. Both are predictable rather than repeatable.

PTARM[ref2] achieves time repeatability via a four-thread interleaved five-stage
pipeline that eliminates data and control hazards between threads. For a single
thread, each instruction takes a fixed number of thread cycles (e.g., one for
arithmetic, four processor cycles with round-robin scheduling). However, PTARM
has low single-thread throughput and requires at least four active threads.
FlexPRET[ref12] extends PTARM to mixed-criticality systems.

Sophon[ref1] proposes a single-cycle RV32I core with no pipeline; fetch, decode,
and execute complete in one cycle (load/store take two cycles). Without branch
prediction and caches, its execution time is fully deterministic. Through a
custom EEI interface, Sophon implements a fGPIO instruction (1-cycle GPIO) and a
snapreg instruction (hardware save/restore of 32 registers). Interrupt latency
in vectoring mode is 3 cycles. Sophon occupies a minimum area of 28.6 kGE,
making it the most advanced real-time RISC-V core to date. Our FX-RV32 further
reduces interrupt latency to 2 cycles and uses the standard RV32I without custom
instructions. Its baseline area is only 24.9 kGE (with the optional shadow
register extension increasing area to 32.4 kGE), offering a competitive
lightweight profile while providing lower interrupt latency.

### Control Latency

#### GPIO Latency

Traditional GPIO controllers attached via on-chip buses incur multi-cycle store-
to-pin delays; the APB GPIO in OpenE902 requires 3 cycles[ref30]. TI's PRU
couples GPIO directly to the core via dedicated registers at 5 ns toggling, but
is proprietary. GPIOCP[ref10] enables cycle-accurate GPIO through a programmable
coprocessor; Sophon's fGPIO[ref1] encodes GPIO access as a custom instruction
for 1-cycle control but requires ISA extension. Zhao et al.[ref11] improved
multi-core GPIO timing accuracy via scheduling. Our FX-RV32 uses memory-mapped
GPIO registers with a single-cycle bus arbiter: writing GPIO_OUT updates the pin
on the next clock edge, giving a fixed 1-cycle latency without custom
instructions.

#### Interrupt Latency

Interrupt latency is critical for hard real-time systems. Standard RISC-V uses
the Platform Level Interrupt Controller (PLIC)[ref17] with a shared entry
requiring software dispatch. The Core Local Interrupt Controller (CLIC)
extension[ref18] supports hardware vectoring and interrupt level preemption, but
software register save/restore overhead remains.

To reduce this overhead, Balas and Benini[ref6] showed that the application
binary interface significantly affects CV32E40P's latency; Mao et al.[ref5]
designed hardware stacking to push 10 caller-saved registers onto a memory stack
in parallel with vector fetch, achieving 13-cycle latency. Other optimizations
include fast interrupt controllers[ref26], hardware machine-mode external
interrupt implementations[ref27], and FCFS PLIC arbitration[ref28]. In FX-RV32,
we combine a fully hardened interrupt path with a shadow register extension to
achieve 2-cycle interrupt latency; details are in Section III.

### Hardware Acceleration Techniques: Shadow Registers and Interrupt Pipeline

Shadow registers provide an additional physical register set for fast context
save/restore without memory access, unlike ARM Cortex-M NVIC's automatic
stacking. A dedicated interrupt pipeline decouples interrupt response from the
main pipeline, ensuring minimal control transfer latency. FX-RV32 tightly
couples both techniques: the interrupt pipeline coordinates acceptance and
vectoring, while shadow registers complete the parallel save of x1–x31 in one
cycle, achieving zero software overhead context switching. Details appear in
Section III.

## Microarchitecture of FX-RV32

FX-RV32 aims to provide time-repeatable execution, ultra-low interrupt latency,
and lightweight hardware. Its baseline core implements the RV32I base integer
instruction set (no multiply/divide, no compressed, no floating point), uses a
five-stage in-order pipeline and a Harvard architecture, and completely abandons
caches and branch prediction. By automatically saving the machine mode critical
state (mepc, mcause, mstatus) and completing the vector jump in hardware, the
interrupt latency is fixed at 2 clock cycles, independent of how general-purpose
registers are saved. In 55 nm CMOS, the baseline core (excluding data RAM)
occupies only 24.9 kGE. To further accelerate general-purpose register saving
inside the interrupt service routine (ISR), we designed an optional shadow
register extension that compresses the save time of the 31 general-purpose
registers x1–x31 from tens of software instructions down to 1 clock cycle. This
extension adds 7.46 kGE, bringing the total core area to 32.4 kGE. Fig. 1 shows
the overall architecture.

FX-RV32 has three compile-time Verilog parameters. SHADOW_EN is defined in three
modules (id_top.v, regfile.v, interrupt_pipeline.v) with a default of 1. Setting

The core consists of the following main modules: - Instruction ROM :
Single-cycle tightly coupled read-only memory storing program code. Read latency
fixed to 1 cycle. - Data RAM : Single-port SRAM storing data. Write
latched on rising clock edge, read is combinatorial output, latency fixed to 1
cycle. - Five-stage pipeline: IF (fetch), ID (decode), EX (execute), MEM (memory
access), WB (write back). Pipeline registers separate stages, all positive edge
triggered. - Register file : 32×32-bit general-purpose registers
(x0–x31). When SHADOW_EN=1, an additional 31 shadow registers
(shadow[1]–shadow[31]) are instantiated. - Interrupt controller : Receives
external interrupts, supports vectoring mode and software-programmable priority arbitration.
Contains interrupt pending (IP), interrupt enable (IE), interrupt priority
(IPR), and interrupt threshold (ITHR) registers. - Interrupt pipeline
: Finite state machine independent of the main pipeline,
responsible for interrupt detection, safety point checking, generating shadow
save/restore signals, and issuing selective pipeline flushes. - Bus arbiter
: Purely combinatorial address decoder mapping MEM stage addresses
to data RAM (low address range) or peripherals (high address range). No wait
states; read data returns in the same cycle. Our bus arbiter design was inspired
by the open source on-chip communication platform[ref16]. - Peripherals: GPIO,
UART, timer, SPI, I2C, each implemented with a minimal feature set; control
registers directly mapped to the bus address space, read/write latency 1 cycle.

All combinatorial logic paths (except the bus arbiter) complete within a single
cycle. No cache logic or speculative mechanisms are present, guaranteeing full
timing determinism.

### Five-Stage Pipeline Design

We adopt a five-stage in-order pipeline; each stage completes a fixed function
in one clock cycle. All pipeline registers are positive edge-triggered; no
negative-edge clocking or gated clocks are employed in the design.

1) Instruction Fetch (IF): Fig. 2 shows the microarchitecture of the IF stage. The core of the instruction fetch stage consists of a
PC register and an instruction ROM interface. After reset, the PC is initialized
to 0x00000000. In each clock cycle, the current value of the PC register is
driven to the address port of the instruction ROM, which returns a 32‑bit
instruction within a single cycle. Meanwhile, the next_pc combinational logic
selects the next PC value according to the following priority: - interrupt
response (intr_take_now, highest priority) - sequential increment during an
accepted interrupt (interrupt_taken) - branch target (branch_taken) - jump
target (jump_taken) - stall (stall) - sequential increment (pc+4, lowest
priority)

The PC register updates to next_pc on the rising clock edge. The fetched 32‑bit
instruction is latched by the IF/ID pipeline register on the next rising clock
edge and is forwarded to the decode stage together with the current PC value. In
the event of a pipeline flush caused by a branch or jump, the IF/ID register is
written with a NOP instruction (0x00000013) to prevent invalid instructions from
propagating further. In the case of an interrupt, the interrupt pipeline
controller handles the operation differently: the PC is already redirected to
the interrupt vector address within the same cycle via the intr_take_now signal,
and the instruction ROM outputs the first instruction of the ISR, which normally
passes through the IF/ID register into the ID stage — therefore, upon an
interrupt, only the ID/EX, EX/MEM, and MEM/WB pipeline registers are flushed to
remove the remaining instructions of the interrupted program, while the IF/ID
register is not flushed, avoiding the insertion of an unnecessary NOP delay.

the instruction decoder, immediate extension unit, register file read ports, and

The decoded results and the read operands are latched into the ID/EX register on

3) Execute (EX): Fig. 4 shows the microarchitecture of the EX stage. The EX stage is the core of data computation and branch
resolution. It contains the ALU, branch/jump unit, and forwarding unit. - ALU:
Supports all RV32I integer operations: addition, subtraction, shifts,
comparisons, logical operations, LUI/AUIPC. All operations are single-cycle

### Harvard Memory Structure and Cache-Less Design

FX-RV32 adopts a Harvard architecture: instruction ROM and data RAM are
physically separated, with independent address and data buses. This separation
allows instruction fetch (IF stage) and data memory access (MEM stage) to
proceed in parallel, avoiding the conflict of instruction and data sharing a
single bus in von Neumann architectures. Fig. 7 illustrates the Harvard architecture and bus arbiter connection. Moreover, because there are no caches,
every memory access has a fixed latency: - Instruction ROM access: 1 cycle -
Data RAM access: 1 cycle - Peripheral access: 1 cycle

This design not only simplifies WCET analysis but also eliminates execution time
jitter caused by cache hits/misses. All memory and peripheral accesses are
independent of history; thus, program execution time depends only on the
instruction sequence and input data, not on internal processor state.

All branches (BEQ, BNE, BLT, BGE, BLTU, BGEU) and unconditional jumps (JAL,
JALR) are resolved in the EX stage. When a branch is taken or a jump occurs, the
EX stage asserts branch_taken or jump_taken and sends the target address to the
IF stage. The hazard detection unit immediately asserts flush_if and flush_id to
clear the two prefetched instructions. In the next cycle, the IF stage fetches
from the target address. - No load-use hazard: Branch penalty is fixed at 2
cycles. - Load-use hazard: If the branch's source register depends on a load
that has not yet been written back, a stall cycle is inserted, making the
penalty 3 cycles.

Because all dependencies are statically analyzable at compile time, program
execution time can still be precisely computed.

### Interrupt Mechanism and Shadow Registers

The interrupt scheme of FX-RV32 is illustrated in Fig. 8. To achieve a good
balance between low latency and hardware complexity, the processor implements
the machine mode interrupt handling mechanism defined by the RISC-V privileged
architecture specification. It supports three interrupt input lines (software,
timer, and external) and introduces a hardware
shadow register extension that performs context saving in parallel with the
interrupt response — within the same 2 clock cycles — and restores context in a
single cycle upon MRET.

[Figure]

*Caption: Overall block diagram of the interrupt system.*

Each interrupt source is managed by a priority encoder inside the interrupt
controller. The priority encoder combinatorially evaluates, in every clock
cycle, the interrupt status from the mie (interrupt enable) and mip (interrupt
pending) registers together with the global interrupt enable bit mstatus.MIE.
When multiple interrupts are pending simultaneously, the encoder selects the
highest priority one according to a programmable priority order (default: MEI, ID=11 > MTI, ID=7 > MSI, ID=3), and outputs an interrupt
pending flag together with the corresponding exception cause code. The interrupt
vectoring mode is selected by the lower two bits (MODE) of the mtvec register.
When MODE=00 (direct mode), all interrupts jump to a single entry address
specified by mtvec; when MODE=01 (vectored mode), the hardware automatically
computes the entry address as handler_addr = BASE + (cause[4:0] << 2), where
each vector slot occupies 4 bytes, just enough to hold one unconditional jump
instruction.

Interrupt acceptance is coordinated by the interrupt pipeline controller
(interrupt_pipeline). This module is a two-state finite state machine
maintaining two internal flags: interrupt_accepted and interrupt_processed. Its
core design goal is to achieve a constant 2-cycle interrupt latency – regardless
of the pipeline state (whether the EX stage contains a branch/jump or the MEM
stage has an unfinished bus read), the first instruction of the ISR is always
executed after exactly 2 clock cycles. This feature relies on three key design
choices:

First, removal of blocking conditions on the EX and MEM stages that are present
in conventional designs. The interrupt no longer waits for a branch/jump in the
EX stage to complete, nor does it wait for a load in the MEM stage to finish.
Instead, the priority chain of the fetch unit's next_pc selection (interrupt >
branch > jump > stall > pc+4) guarantees that when an interrupt arrives, the PC
directly jumps to the interrupt vector address, while branch and jump targets
are automatically ignored.

Second, introduction of the bus_ready signal to decouple pending loads in the
MEM stage. When a load is present in the MEM stage and bus_ready = 0 (access not
yet completed), the hardware deasserts the bus request signal (bus_re_o),
causing the peripheral to abort the transaction; the mepc is set to point to the
load instruction itself, so that the instruction will be re-executed after MRET.
When the load in the MEM stage has already completed (bus_ready = 1), the data
flows normally into the WB stage and is written to the register file, while mepc
points to the next instruction after the load (mem_pc + 4), avoiding duplicate
execution of a completed instruction.

Third, selective pipeline flush upon interrupt acceptance: the hazard unit
asserts intr_flush_if (which is used to stall the IF stage, preventing the PC
from advancing during the interrupt sequence) and intr_flush_ex/mem/wb to flush
the ID/EX, EX/MEM, and MEM/WB pipeline registers with NOPs. Crucially,
intr_flush_id is not asserted — the PC has already been redirected to the
interrupt vector address by the intr_take_now signal in the same cycle that the
interrupt is accepted, and the instruction ROM consequently outputs the first
instruction of the ISR. This instruction passes through the IF/ID register
normally into the ID stage, avoiding the insertion of an unnecessary NOP that
would increase interrupt latency by one cycle. This design ensures that partial
results of the interrupted program (residing in ID/EX and later stages) are not
leaked into the execution context of the ISR, while simultaneously allowing the
first ISR instruction to enter the pipeline without delay.

To eliminate the software overhead of saving/restoring the register file context
inside the interrupt service routine, FX-RV32 integrates a set of hardware
shadow registers inside the register file – 31 additional 32-bit registers,
corresponding to x1–x31 (x0 is hardwired to 0 and needs no saving). Fig. 9 illustrates the internal structure of the register file including the shadow registers.

The operation of the shadow registers is tightly synchronised with the state
transitions of the interrupt pipeline controller. In the cycle immediately after
an interrupt is accepted (the rising edge T1↑), after the pipeline flush has
completed and all in-flight instructions that have not yet reached the write
back stage (i.e., those already in the ID, EX, or MEM stage) have finished their
register write back, the shadow_save signal is asserted as a single-cycle pulse,
capturing a parallel snapshot of the complete register state of x1–x31 into the
shadow registers. When an MRET instruction reaches the EX stage, the interrupt
pipeline controller detects the id_ex_mret signal and triggers a shadow_restore
pulse. On the next rising clock edge, the entire contents of the shadow
registers are written back in parallel to x1–x31. At the same time, the
interrupt_processed flag is cleared and mstatus.MIE is restored, re-enabling
interrupt response. The write back priority of the register file is designed as
shadow_restore (highest) > normal WB write > shadow_save (lowest). This priority
order guarantees that during a restore, any modifications made by the ISR to the
registers are completely overwritten; during a save, the write back data of
instructions that are still in flight but have already produced a result (e.g.,
a bus read that completed in the MEM stage) is written to the register file
before the snapshot is taken, ensuring that the shadow registers hold a truly
complete register state.

The shadow register mechanism is completely transparent to software; the
interrupt service routine does not need any manual push/pop operations. The key
value of shadow registers is that the hardware save of all general-purpose
registers takes only 1 clock cycle (the parallel snapshot triggered by the
single-cycle shadow_save pulse upon interrupt entry), and the hardware restore
also takes only 1 clock cycle (the parallel write back triggered by the single-
cycle shadow_restore pulse upon MRET execution). In contrast, a conventional
software stack-based save/restore scheme requires one store instruction for each
register save and one load instruction for each register restore. Hence, saving
and restoring all 31 writable registers would demand 62 memory access
instructions, incurring a significant time overhead. For the complete round-trip
interrupt latency: interrupt entry takes 2 clock cycles, while MRET return takes
3 clock cycles (MRET in EX stage → PC jumps to mepc and fetches → shadow
register restore completes, and the first instruction of the interrupted program
enters the ID stage with restored register values).

The shadow register feature is controlled by a compile-time parameter SHADOW_EN,
which is independently defined in the id_top and interrupt_pipeline modules.
When SHADOW_EN = 0, the shadow registers and their control logic are completely
optimised away by the synthesis tool, and the ISR must manually save/restore
context using software stack operations. This configuration is suitable for
area-sensitive applications or those requiring interrupt nesting. The current
design provides only a single set of shadow registers and therefore does not
support interrupt nesting. If nested interrupts are required, the shadow
register file can be extended to a multi-bank stack structure.

The CSR register file implements all core CSRs required for machine mode,
including mstatus, mtvec, mepc, mcause, mie, mip, mscratch, mtval, as well as
the two 64-bit hardware performance counters mcycle and minstret. A key design
feature is its dual-port write architecture: one write port is dedicated to CSR
instructions (CSRRW, CSRRS, CSRRRC and their immediate variants), while the
other write port is reserved for concurrent updates to mepc, mcause, and mstatus
by the interrupt pipeline controller during interrupt response. The two ports
may be active simultaneously within the same cycle without conflict.

Under the interrupt system architecture described above, the total latency from
interrupt acceptance by the core to the execution of the first ISR instruction
is constant at 2 clock cycles, which at an operating frequency of 200 MHz
corresponds to only 10 ns. The hardware shadow registers compress both context
saving and restoring to 1 clock cycle each, reducing the complete round-trip
interrupt overhead (from the last instruction of the interrupted program to the
first instruction of the ISR, and then from MRET to the first instruction after
return) to its theoretical minimum.

### GPIO and Peripheral Interface

The GPIO module is memory-mapped, with registers including: GPIO_OUT (output
data), GPIO_OE (output enable), GPIO_IN (input data), GPIO_IE (interrupt
enable), GPIO_EDGE (trigger edge/level), GPIO_IF (interrupt flag). After writing
to GPIO_OUT, the pin updates on the next rising clock edge, giving a fixed
output latency of 1 cycle. All peripheral accesses are single-cycle, without
FIFOs or DMA, further enhancing system determinism.

For comparison, Sophon[ref1] achieves 1-cycle GPIO control through a custom
fGPIO instruction encoded via its EEI interface, which requires ISA extension
and increases software porting complexity. OpenE902 employs an APB-attached GPIO
controller[ref30]; a store instruction must traverse the APB bus, incurring a
3-cycle store-to-pin delay, limiting the maximum GPIO toggling frequency to
one-sixth of the CPU frequency. FX-RV32 achieves the same 1-cycle latency as
Sophon's fGPIO using only standard RV32I store instructions, because its bus
arbiter performs purely combinatorial address decoding without wait states and
the GPIO\_OUT register is updated directly on the next clock edge. No custom
instruction or ISA modification is required, preserving full software
compatibility. Fig. 11 compares the GPIO write timing of the three processors.

## Experimental Evaluation

### Experimental Setup

We conducted comprehensive synthesis and FPGA verification of the FX-RV32 core
to evaluate its hardware overhead, timing determinism, interrupt latency, and
GPIO control precision. All experiments were carried out on a uniform 55 nm
process library and a Xilinx Kintex-7 FPGA platform, and compared against
multiple open-source RISC-V cores.

#### Synthesis and ASIC Implementation

We used the SMIC 55LL HD RVT 55 nm CMOS standard cell library and Synopsys
Design Compiler 2019.12 for synthesis, with the compile_ultra optimization
strategy. Two configuration variants of FX-RV32 were synthesized: a baseline
version (RV32I only, shadow registers disabled) and a full version (RV32I +
hardware shadow registers), in order to quantify the area cost of the shadow
register module. For comparison, the open-source RISC-V core OpenE902 (open-
sourced by T-Head, RV32EMC, two-stage in-order pipeline, CLIC vectored interrupt
controller, hardware multiplier/divider) was synthesized under the same library
and constraints. The area and critical path data for the six configuration
variants of Sophon[ref1] (baseline RV32I, with CLIC extension, and with EEI
external interfaces scaled from 4 to 8/16/32 channels) are taken directly from
the literature. Area was obtained from cell_area and converted to NAND2 gate
equivalents (1 GE ≈ 1.12 µm²); the
critical path was obtained from report_timing. The synthesis results are
summarized in Fig. 13.

[Figure]

#### FPGA Prototyping

FX-RV32 was deployed as a soft core on a Xilinx Kintex-7 XC7K325T-2FFG900C FPGA
(Genesys 2 board). FPGA synthesis used Xilinx Vivado 2022.1; the operating
frequency was set to 200 MHz (using the LVDS clock input on the Genesys 2
board). The core logic (excluding data RAM) occupied about 2056 LUTs and 1913
registers. Instruction ROM and data RAM were implemented as Block RAMs (4 KB
each).

#### Comparison Targets
- OpenE902: A two-stage in-order pipeline RISC-V core open-sourced by T-Head Semiconductor, targeting low-cost embedded applications. It supports the RV32EMC instruction set (RV32E base integer + hardware multiplier/divider + compressed instructions) and M/U privilege modes, and is equipped with a CLIC vectored interrupt controller. We used its default configuration (RV32EMC, CLIC vector mode enabled) and synthesized it under the same library for a fair area and timing comparison. Its interrupt latency (hardware vectoring mode) and GPIO control latency (APB-based) also serve as a reference baseline for evaluating the real-time performance of FX-RV32.
- Sophon[ref1]: A single-cycle RV32I core featuring custom fGPIO instructions (single-cycle GPIO control) and an EEI external interface, with CLIC extension support. The literature reports interrupt latency under three processing modes: 3 cycles in hardware vectoring + inline handler mode (2 cycles to load the vector address from L1 data memory, 1 cycle for control transfer); 7 cycles in non-vectored mode with the custom snapreg instruction (snapreg performs single-cycle context save/restore); and 39 cycles in non-vectored mode with a C ABI handler (16 caller-saved registers are preserved sequentially via software store/load instructions). The baseline area at 55 nm is approximately 28.6 kGE. Sophon represents the most prominent recent work in low interrupt latency and deterministic response; its area/frequency data (across multiple configurations) and interrupt latency data are taken directly from the literature for systematic comparison with FX-RV32 along three dimensions: interrupt latency, GPIO latency, and hardware overhead.
- PicoRV32: An open-source RV32I core designed by Clifford Wolf, targeting minimal area. It employs a multi-cycle state machine implementation without pipelining, caches, or branch prediction. Its interrupt mechanism is a non-standard fixed-entry scheme — there are no standard CSRs such as mtvec, mcause, or mepc, nor a vector table. Upon an interrupt trigger, the hardware jumps to the hard-coded address PROGADDR_IRQ (0x10), and the ISR manages context save/restore and interrupt source identification through custom instructions (getq/setq/retirq/maskirq). Since the multi-cycle state machine must complete the current instruction before responding to an interrupt, its interrupt latency exhibits significant variation — 7 cycles in the best case, and up to several tens of cycles in the worst case (depending on the type of the interrupted instruction; division/multiplication instructions require up to 40 cycles to complete). PicoRV32 serves as the area floor reference among lightweight RISC-V cores; its inclusion in the comparison illustrates where FX-RV32 sits in the area–real-time performance trade-off spectrum, achieving deterministic low-latency interrupt performance at a moderate area cost.
- Fast Interrupt (CLIC hardware stacking scheme): Mao et al.[ref5] proposed a hardware stacking interrupt architecture based on the RISC-V CLIC specification. Upon interrupt response, the hardware automatically pushes 10 caller-saved registers onto an internal stack, eliminating the software save instruction sequence in the ISR. The reported interrupt latency is 13 clock cycles, of which 11 cycles are consumed by the hardware stacking operation. This scheme represents the performance level achievable by the design approach of "hardware-accelerated context saving without shadow registers," and serves to evaluate the latency advantage of full parallel shadow register saving over serial hardware stacking.
- CV32E40P[ref6]: An open-source four-stage in-order pipeline RISC-V core (formerly RI5CY from the PULP platform), supporting the RV32IMFC instruction set and equipped with a custom vectored interrupt controller. Its interrupt latency is 33 cycles under the regular ABI, reduced to 24 cycles after EABI optimization — in both modes, the most time-consuming operation is software context save/restore executed instruction by instruction within the ISR. CV32E40P represents the performance upper bound of interrupt architectures without any hardware context acceleration, and its latency data inversely validates the effectiveness of hardware shadow registers in eliminating software save overhead.

#### Measurement Methods
- Area/frequency: Gate count derived from DC's cell_area report (1 GE = 1.12 µm²); critical path from report_timing.
- Deterministic tests: Test programs were compiled to RV32I machine code, written into instruction ROM, and the mcycle CSR was read via a JTAG debug interface. Each test was run 40 times, and cycle counts were recorded.
- Interrupt latency: A GPIO pin generated an external interrupt (rising edge). The number of clock cycles from interrupt acceptance by the core to the execution of the first ISR instruction was measured by observing the PC bus and interrupt signals in simulation waveforms.
- GPIO output latency: A loop continuously wrote alternating values to the GPIO_OUT register (0xFFFFFFFF then 0x00000000). The delay from the store instruction to the pin level change was measured in simulation waveforms.

### Hardware Overhead and Frequency

Fig. 13 summarizes the area and critical path of all synthesis configurations.
The FX-RV32 baseline version (RV32I only, shadow registers disabled) achieves a
synthesis area of 27,879 µm² (approximately 24.9 kGE) at 55 nm, with a critical
path of 4.89 ns, corresponding to a maximum frequency of 204 MHz. Among RISC-V
cores that feature pipelining or interrupt acceleration, this is the smallest
area — even slightly lower than the Sophon RV32I baseline (32,116 µm²,
approximately 28.7 kGE). This demonstrates that the five-stage pipeline design
is not inferior to single-cycle architectures in area efficiency: the additional
flip-flop overhead introduced by pipeline registers (approximately 2 kGE) is
offset by the larger combinational logic paths in single-cycle designs. The only
area floor below this is the PicoRV32 baseline version (20,808 µm²), but it
employs a non-pipelined multi-cycle state machine implementation with CPI ≈ 4,
placing the two in different performance classes.

With hardware shadow registers enabled (full version), the area increases to
36,252 µm² (approximately 32.4 kGE), an increase of about 8,360 µm² (+30%).
These 8,360 µm² represent the hardware cost of 31 32-bit shadow registers and
their read/write control logic. Notably, the introduction of shadow registers
has virtually no impact on the critical path (4.89 ns → 4.88 ns), because the
shadow register save/restore operations do not lie on the critical path from the
ALU to the data RAM address decoding — they constitute a parallel internal path
within the register file. This means the shadow registers trade pure area for
single-cycle context save capability without sacrificing frequency.

OpenE902, synthesized under the same library and constraints, occupies an area
of 83,924 µm² (approximately 74.9 kGE), which is 3.0× the FX-RV32 baseline and
2.3× the full version. Its critical path of 5.63 ns (maximum 178 MHz) is
approximately 15% longer than that of FX-RV32. The additional area of OpenE902
primarily stems from the RV32C compressed instruction decoder, the hardware
multiplier/divider unit, and the more complex CLIC vectored interrupt controller
— features that significantly increase hardware overhead while enhancing
functionality. Sophon's six configuration variants[ref1] are included in Fig. 13
for reference. Notably, Sophon's baseline RV32I configuration (28.7 kGE) does
not provide vectored interrupt or fast GPIO — these require the CLIC and EEI
extensions, which together raise the area to 37,939 µm² (+18.1%) at minimum, and
up to 70,737 µm² (+86.4%) with 32 EEI channels. In contrast, FX-RV32 delivers
2-cycle vectored interrupt and 1-cycle GPIO at its baseline area of 24.9 kGE
using only standard RV32I instructions, without any ISA extensions or additional
interface logic. Among all compared configurations, FX-RV32 also achieves the
best critical path (4.88–4.89 ns).

PicoRV32's two configuration variants represent two classic operating points of
the area-first design strategy. Its baseline version occupies only 20,808 µm²
(approximately 18.6 kGE) with a critical path of 4.90 ns — the smallest among
all synthesized samples, confirming its position as the area floor for RISC-V
soft cores, but this comes at the cost of low instruction throughput (CPI ≈ 4)
due to its multi-cycle state machine implementation, and non-deterministic
interrupt response (latency varying from 7 to tens of cycles). Its full version
(with IRQ, QREGS, timer, and all interrupt features enabled) grows to 33,086 µm²
(approximately 29.5 kGE, +59%), with a critical path of 4.95 ns. Remarkably, the
FX-RV32 baseline (27,879 µm², 24.9 kGE) is 16% smaller than the PicoRV32 full
version, yet provides a five-stage pipeline with instruction-level parallelism
(CPI approaching 1) and fully deterministic execution behavior. Even with shadow
registers enabled (36,252 µm², 32.4 kGE), FX-RV32 is only about 10% larger than
PicoRV32 full, while additionally offering single-cycle hardware context
save/restore capability — compressing the save/restore of 31 registers in the
ISR from tens of software store/load instructions to a single clock cycle.
Coupled with FX-RV32's fixed 2-cycle interrupt response latency (guaranteed
independently by the interrupt pipeline architecture), this stands in stark
contrast to PicoRV32's 7-to-tens-of-cycles interrupt latency fluctuation.

Overall, the FX-RV32 baseline achieves the best critical path (4.88–4.89 ns)
among all compared cores at a mere 24.9 kGE. The shadow register's 7.46 kGE area
increment (+30%) trades pure area for reducing context save/restore overhead to
a single clock cycle, without sacrificing frequency. FX-RV32's 2-cycle interrupt
response latency is independently guaranteed by the interrupt pipeline's
unconditional acceptance mechanism and hardware direct vector address
computation. In horizontal comparison with the PicoRV32 full version, FX-RV32
provides significantly superior instruction throughput, interrupt response speed,
and execution determinism at comparable or even smaller area — precisely the
combination of qualities that is most valuable in embedded real-time scenarios.

### Interrupt Latency

To evaluate the interrupt response behavior of FX-RV32, we measured the number
of clock cycles from when an interrupt request is accepted by the core to when
the first instruction of the interrupt handler is executed. The measurement
covers the hardware path of interrupt response — interrupt arbitration, vector
address computation, pipeline flush, and PC redirection — and excludes context
save/restore; in FX-RV32, the latter is performed in parallel by hardware shadow
registers within a single cycle. The test used a GPIO external interrupt (ID
11), mtvec in vectored mode (MODE = 01), base address 0x200 (vector entry: 0x200
+ 11 × 4 = 0x22C). The first ISR instruction was a nop; the ISR ended with mret.
The simulation waveforms captured intr_external_i, interrupt_o, shadow_save, and the PC
bus. For comparison, we measured the interrupt latency of OpenE902 in CLIC
hardware vectoring mode on the same FPGA platform (identical interrupt source
and triggering method). The interrupt latency data for Sophon[ref1], Fast
Interrupt (CLIC hardware stacking scheme)[ref5], and CV32E40P[ref6] were taken
from the respective literature; the interrupt latency of PicoRV32 was obtained
by injecting GPIO interrupts and observing waveforms in its Verilator simulation
environment. The experimental results are summarized in Fig.[Fig].

FX-RV32 achieves a fixed 2-cycle interrupt latency. In cycle 1, the interrupt
controller completes priority arbitration and asserts interrupt_o; driven by the
intr_take_now combinational signal, the PC is redirected to the handler address
in the same cycle. The interrupt pipeline unconditionally accepts the request
regardless of EX-stage branch/jump or MEM-stage load status (see Section III),
and shadow_save is asserted, triggering parallel save of x1–x31. In cycle 2,
the first ISR instruction passes through the ID stage for decoding; at the end
of this cycle it is latched into the ID/EX register and enters the execute
stage.

[Figure]

*Caption: Interrupt latency comparison among RISC-V CPUs: Sophon (C ABI 39 cycles / snapreg 7 cycles / hardware vectoring 3 cycles), OpenE902 (hardware vectoring 9 cycles), Fast Interrupt (hardware stacking 13 cycles), CV32E40P (EABI 24 cycles / regular ABI 33 cycles), PicoRV32 (non-vectored custom retirq, fastest 7 cycles), FX-RV32 (hardware vectoring 2 cycles).*

Sophon[ref1] achieves an interrupt latency of 3 cycles in hardware vectoring
mode, the closest to FX-RV32 among all compared cores. Its 3-cycle composition
is: 2 cycles to load the vector table address from L1 data memory, and 1 cycle
to transfer control to the interrupt handler. The key reason FX-RV32 is able to
further compress the latency to 2 cycles lies in the different method of vector
address computation — the interrupt controller of FX-RV32 directly computes
mtvec + cause × 4 using pure combinational logic, requiring no memory access to
load the vector table, thereby merging address generation and PC redirection
into a single cycle. In non-vectored mode, the interrupt latency of Sophon
increases substantially: 39 cycles when using a C ABI handler — time mainly
consumed by sequentially saving 16 caller-saved registers using store
instructions; even with the custom snapreg instruction compressing context save
to a single cycle, the total latency remains 7 cycles (1 cycle for snapshot save
+ 6 cycles for the remaining hardware path).

OpenE902 exhibits an interrupt latency of 9 clock cycles in CLIC hardware
vectoring mode. Its two-stage pipeline architecture requires a complete pipeline
flush operation upon interrupt response, and the CLIC vector table loading
latency exceeds that of Sophon's single-cycle design, ultimately resulting in an
interrupt latency approximately 3× that of Sophon's hardware vectoring mode and
4.5× that of FX-RV32. Fast Interrupt (CLIC hardware stacking scheme)[ref5] has
an interrupt latency of 13 clock cycles, of which 11 cycles are consumed by the
hardware serial stacking operation to save 10 caller-saved registers — a scheme
that replaces the software store instruction sequence with a hardware state
machine operating in serial fashion. Although this avoids instruction
fetch/decode overhead, its serial nature prevents it from competing with the
fully parallel save capability of shadow registers (31 registers saved in 1
cycle). CV32E40P[ref6] exhibits an interrupt latency of 33 cycles under the
regular ABI, reduced to 24 cycles after EABI optimization — in both modes, the
most time-consuming operation is the software context save/restore executed
instruction by instruction within the ISR. CV32E40P represents the performance
upper bound of interrupt architectures without any hardware context
acceleration, and its latency data inversely validates the effectiveness of
hardware shadow registers in eliminating software save overhead.

PicoRV32[ref29] achieves 7 cycles in the best case, but this figure requires
careful interpretation. PicoRV32 uses a non-standard interrupt mechanism without
RISC-V CSRs (mtvec, mcause, mepc) or a vector table; interrupts jump to a hard-
coded address (0x10), and the ISR must query the interrupt source via custom
getq instructions. Its 7 cycles cover only the minimal "detect→complete→jump"
path plus basic context saving via getq/setq, excluding vector dispatching.
Moreover, PicoRV32's multi-cycle state machine introduces significant non-
determinism: simple ALU instructions respond in 7 cycles, but multi-cycle
instructions such as division (up to 40 cycles) or multiplication (up to 8
cycles) must complete before interrupt handling begins, causing latency
variation of tens of cycles. In contrast, FX-RV32's 2-cycle latency is strictly
deterministic and independent of instruction type, and its standard RISC-V
vectored interrupt mode enables hardware-automatic vector table indexing and
dispatch without software intervention.

In summary, among these RISC-V cores, FX-RV32 achieves the lowest interrupt
latency of 2 clock cycles — a 33% reduction compared to Sophon's hardware
vectoring mode (3 cycles). Critically, this 2-cycle figure already encompasses
the full context save of all 31 general-purpose registers: the shadow registers
fire in parallel with the interrupt response, completing the save within the
same 2 cycles. Sophon's 3 cycles also include context save (via the custom
snapreg instruction), while all other compared cores require substantially more
cycles for software save/restore. This latency is strictly deterministic,
independent of the interrupted instruction type or pipeline state, fully
demonstrating the advantage of the synergistic design of the interrupt pipeline
and shadow register mechanisms.

## Conclusion

This paper presented FX-RV32, a lightweight, deterministic RISC-V processor for
hard real-time systems. It employs a five-stage in-order pipeline with a cache-
less Harvard architecture, achieving a fixed 2-cycle interrupt latency—the
lowest among open-source RISC-V cores—and 1-cycle GPIO output via standard RV32I
instructions. At 55 nm, the baseline core occupies 24.9 kGE (32.4 kGE with
optional shadow registers that save context in parallel with interrupt response and restore in a single cycle upon MRET). Eight
deterministic benchmarks exhibited zero cycle-count jitter across 40 runs each.
FX-RV32 improves interrupt latency by 33% over Sophon while maintaining ISA
compatibility. Future work includes compressed instruction support, multi-bank
shadow registers for interrupt nesting, and memory protection integration.

The FX-RV32 implementation is open source and can be freely accessed from
https://github.com/News111234/FX-RV32.
