# 影子寄存器硬件上下文保存/恢复 功能说明

## 1. 概述

本功能为 FX-RV32 RISC-V CPU 核心添加**硬件级上下文自动保存/恢复机制**。当中断发生时，硬件自动将 x1-x31 通用寄存器的值保存到影子寄存器中；当中断处理程序执行 MRET 指令返回时，硬件自动将影子寄存器中的值恢复到 x1-x31，使被中断的程序无感恢复执行。

## 2. 设计动机

标准的 RISC-V 中断处理流程中，中断服务程序（ISR）需要用软件指令手动保存和恢复寄存器上下文（通常通过栈操作），这不仅增加了中断延迟，也增加了代码复杂度和功耗。本功能将上下文保存/恢复完全硬件化，实现：

- **零开销上下文切换**：中断进入时硬件自动快照寄存器，无需软件逐条 push
- **确定性中断延迟**：保存操作固定在一个时钟周期内完成
- **透明恢复**：MRET 时硬件自动恢复，被中断程序完全无感

## 3. 原理与执行流程

### 3.1 架构框图

```
                    +-----------+
  中断请求 ------->| interrupt |---- shadow_save -----> +------------+
                    | pipeline  |---- shadow_restore --> |  regfile   |
   id_ex_mret ----->|           |                        |            |
                    +-----------+                        | x0 - x31   |
                                                         | shadow[1:31]|
                                                         +------------+
```

### 3.2 影子寄存器保存流程（中断进入）

```
步骤1: 中断条件满足
  - interrupt_controller 检测到中断挂起
  - interrupt_pipeline 判断流水线状态: EX无分支/跳转, MEM无load
  - 断言 interrupt_accepted = 1

步骤2: 流水线冲刷与CSR更新（中断进入的第1个时钟周期）
  - PC 重定向到 mtvec（中断向量表入口）
  - 流水线所有阶段被冲刷（插入NOP）
  - CSR写入: mepc = 中断PC, mcause = 中断原因
  - mstatus更新: MIE清零(关中断), MPIE=旧MIE, MPP=3

步骤3: 影子寄存器保存（中断进入的第2个时钟周期）
  - 此时流水线已完全冲刷, 且所有飞行中指令的WB写入已完成
  - shadow_save 信号拉高一周期
  - 在时钟上升沿: shadow_registers[i] <= registers[i]  (i = 1..31)
  - x0 无需保存（硬连线为0）

步骤4: 中断处理程序开始执行
  - PC = mtvec, 第一条 handler 指令被取指
  - handler 可自由使用 x1-x31, 原始值已安全保存在影子寄存器中
```

### 3.3 影子寄存器恢复流程（MRET 返回）

```
步骤1: MRET 进入 EX 阶段
  - decoder 识别 MRET 指令 (opcode=0x73, funct12=0x302)
  - ID/EX 流水线寄存器传递 mret 标志
  - id_ex_mret 信号拉高

步骤2: 影子寄存器恢复与流水线操作（MRET在EX的周期）
  - ex_top 将 jump_target 设为 mepc
  - jump_taken 拉高, PC 重定向到 mepc
  - 流水线 IF/ID 被冲刷
  - shadow_restore 信号拉高一周期
  - 在时钟上升沿: registers[i] <= shadow_registers[i]  (i = 1..31)
  - shadow_restore 优先级最高, 会覆盖同周期的任何WB写操作

步骤3: interrupt_pipeline 状态更新
  - interrupt_processed 清零, 允许再次响应中断
  - mstatus 恢复: MIE = MPIE（重新开中断）

步骤4: 被中断程序恢复执行
  - PC = mepc, 原程序的下一条指令被取指
  - 此时 x1-x31 已完全恢复为中断前的值
  - 被中断程序无感继续执行
```

### 3.5 优先级设计

regfile 中的写优先级如下：

| 优先级 | 操作 | 说明 |
|--------|------|------|
| 1（最高） | shadow_restore | 恢复影子寄存器到 x1-x31，覆盖同周期WB写入 |
| 2 | 正常 WB 写 | 通用寄存器写回（we_i && waddr_i != 0） |
| 3（最低） | shadow_save | 保存 x1-x31 到影子寄存器 |

优先级 2 高于 3 的设计确保：中断进入时，先让飞行中的指令完成WB写入，再将完整的寄存器状态保存到影子寄存器。

优先级 1 最高的设计确保：MRET 返回时，被中断程序的寄存器状态完整恢复，中断处理程序中的任何寄存器修改都不会泄漏。

## 5. 使用方法

### 5.1 对程序员透明

本功能**对软件完全透明**。程序员无需：

- 在中断处理程序中手动 push/pop 寄存器
- 调用任何特殊的保存/恢复指令
- 修改链接脚本或启动代码

**唯一要求**：中断处理程序必须以 `mret` 指令结束（这是 RISC-V 标准要求）。

### 5.2 中断处理程序示例（汇编）

```asm
# 中断向量表 (mtvec)
vector_base:
    j timer_handler      # 0x00: 定时器中断
    j external_handler   # 0x04: 外部中断
    # ... 其他中断向量

# 定时器中断处理程序
# 注意: 无需保存/恢复寄存器, 硬件自动完成!
timer_handler:
    # 直接使用寄存器, 无需 push
    li   t0, 0x10000000    # 定时器基地址
    lw   t1, 0(t0)         # 读取定时器状态
    addi t1, t1, 1         # 更新计数
    sw   t1, 0(t0)         # 写回
    # 硬件会在 mret 时自动恢复 x1-x31
    mret
```

### 5.3 中断处理程序示例（C 语言）

```c
// 定时器中断处理程序
// 注意: 无需 __attribute__((interrupt)) 中的寄存器保存
void timer_handler(void) {
    // 直接访问外设寄存器
    volatile uint32_t *timer = (uint32_t *)0x10000000;
    *timer = *timer + 1;
    
    // 函数返回后执行 mret, 硬件自动恢复上下文
}

// 启动代码中的中断向量表
__attribute__((section(".vectors")))
void (*vector_table[])(void) = {
    [0] = default_handler,
    [7] = timer_handler,      // MTI (机器定时器中断)
    [11] = external_handler,  // MEI (机器外部中断)
};
```

### 5.4 RTOS 上下文切换的配合

如果需要在中断中进行任务切换（RTOS 场景），可以这样做：

```asm
switch_handler:
    # 1. 将当前任务的影子寄存器内容读出（通过自定义CSR或内存映射）
    # 2. 将影子寄存器保存到当前任务的TCB
    # 3. 从新任务的TCB加载上下文到影子寄存器
    
    # 注意: 硬件自动保存的寄存器在影子中，
    # 切换任务需要手动操作影子寄存器内容
    mret
```

## 6. 资源消耗

| 资源 | 增加量 | 说明 |
|------|--------|------|
| 寄存器 | 31 × 32 bit = 992 bit | 31个32位影子寄存器 |
| 组合逻辑 | 少量 | 多路选择器用于恢复时的并行写入 |
| 控制信号 | 2 根线 | shadow_save, shadow_restore |
| 时钟频率影响 | 极小 | 关键路径不变（影子寄存器操作在 regfile 内部，不增加外部路径延迟） |

在 FPGA 实现中，992 bit 约合 31 个 LUT（如果使用分布式 RAM）或少量 Block RAM。

## 7. 注意事项与限制

1. **不支持嵌套中断**：当前 CPU 设计中，进入中断后 MIE 被清零，不支持嵌套。如果需要嵌套中断，需要进一步修改 shadow 寄存器为多组（如使用堆栈式影子寄存器）。

2. **x0 不保存**：x0 寄存器硬连线为 0，无需保存。

3. **CSR 寄存器不保存**：影子寄存器机制仅保存通用寄存器 x1-x31。mstatus、mepc、mcause 等 CSR 仍由标准 RISC-V 中断机制管理。

4. **FPU 寄存器不保存**：本 CPU 不支持浮点单元，无 f0-f31 寄存器。

5. **中断处理程序中的子函数调用**：处理程序内调用子函数时，如果子函数修改了 ra(x1) 等寄存器，这些修改会在 MRET 时被影子寄存器恢复覆盖。这通常不是问题，因为中断返回后应该恢复到被中断程序的上下文。

6. **与 YIELD 指令的配合**：YIELD 指令触发软件中断，同样享受硬件上下文保存/恢复的好处。

## 8. 调试建议

在仿真中验证影子寄存器功能时，建议观察以下信号：

- `core_top.shadow_save`：中断进入后应为 1 周期脉冲
- `core_top.shadow_restore`：MRET 进入 EX 后应为 1 周期脉冲
- `regfile.shadow_registers[1]` - `[31]`：中断后应包含中断前的寄存器值
- 中断返回后，x1-x31 的值应与中断前完全相同

波形关键检查点：
1. 中断前记录 x1-x31 的值
2. 中断处理程序修改 x1-x31
3. MRET 执行后，验证 x1-x31 恢复到中断前的值
